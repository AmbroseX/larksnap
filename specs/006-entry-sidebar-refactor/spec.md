# Feature Specification: 交互入口与侧边栏布局重构

**Feature Branch**: `006-entry-sidebar-refactor`
**Created**: 2026-07-13
**Status**: Draft
**Input**: 基于 docs/plans/2026-07-13-交互入口与侧边栏重构.md 生成功能规格：重构 larksnap 扩展的交互入口与侧边栏布局（工具栏图标直开侧边栏并删除 popup、右键菜单按页面类型分组显隐、4 个键盘快捷键、侧边栏改为"当前页上下文区置顶 + 通用工具常驻"的分组布局、统一动作分发层显式传递目标页面、角标反馈机制）

## Clarifications

- Q: popup 是否彻底删除（桥接状态迁到侧边栏 header 的连接状态圆点）？ → A: 彻底删除。点图标直开侧边栏，状态信息由 header 状态点等价承载。
- Q: 飞书页右键菜单放哪些导出项？ → A: MD + PDF + HTML 三项，另加一项"更多导出方式…"引导去侧边栏。样式类动作（小红书/公众号）不进右键。
- Q: 侧边栏通用工具区各分组的默认折叠状态？ → A: 剪藏组默认展开，其余（截图、AI 总结、页面开关）默认折叠；折叠态持久保存，用户改过后以用户为准。
- Q: 「写入飞书文档 / 采集到多维表格」的 UI 入口是否排除在本次范围外？ → A: 排除，作为独立后续计划，本 spec 仅立此存照。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 一次点击到达功能面板 (Priority: P1)

作为 larksnap 用户，我点击浏览器工具栏上的扩展图标时，希望直接进入功能面板（侧边栏）开始干活，而不是先看到一个只显示连接状态的中转弹窗、再点一次按钮才能进入。

**Why this priority**: 工具栏图标是扩展最贵的入口，当前"两次点击"是所有用户每次使用都要付出的成本；且此改动是后续入口体系的地基。

**Independent Test**: 安装扩展后点击工具栏图标，侧边栏直接打开；原弹窗不再出现；原弹窗承载的连接状态信息能在侧边栏内完整找到。

**Acceptance Scenarios**:

1. **Given** 扩展已安装且当前在任意网页, **When** 用户点击工具栏图标, **Then** 侧边栏一步打开，不出现任何中转弹窗。
2. **Given** 本地桥接服务已连接, **When** 用户查看侧边栏顶部, **Then** 看到绿色状态圆点；点击圆点展开一行详情，可看到桥接服务版本，且会话标识可一键复制。
3. **Given** 本地桥接服务未连接, **When** 用户查看侧边栏顶部, **Then** 状态圆点为灰色；连接中为黄色。
4. **Given** 用户从未使用过桥接服务, **When** 使用扩展任何导出/剪藏功能, **Then** 一切正常，状态圆点不干扰主流程。

---

### User Story 2 - 右键菜单覆盖高频动作并按页面类型显隐 (Priority: P1)

作为用户，我在网页上点右键时，希望在一个统一的 LarkSnap 父菜单下看到"当前这类页面"最常用的动作：普通网页看到剪藏/截图/总结，飞书文档页看到导出，而不是所有页面都只有 3 个剪藏项。

**Why this priority**: 右键是"对着页面/选区操作"的天然入口，当前覆盖极不均衡（仅剪藏 3 项），飞书导出、截图、总结全都无法从右键触达。

**Independent Test**: 分别在普通网页、飞书文档页、选中文字状态下点右键，核对 LarkSnap 父菜单下的子项集合与本 spec 定义一致；点击每一项能触发对应动作。

**Acceptance Scenarios**:

1. **Given** 用户在普通网页（非飞书域）, **When** 点右键展开 LarkSnap 菜单, **Then** 可见：整页转 Markdown、整页截图、AI 总结本页、解除复制限制、打开 LarkSnap 面板。
2. **Given** 用户在普通网页选中了一段文字, **When** 点右键, **Then** 额外可见"选区转 Markdown"。
3. **Given** 用户在飞书文档页（含已授权的私有化域）, **When** 点右键展开 LarkSnap 菜单, **Then** 可见：导出为 Markdown、导出为 PDF、导出为 HTML、更多导出方式…（打开侧边栏）。
4. **Given** 用户新授权了一个私有化飞书域名, **When** 在该域的文档页点右键, **Then** 能看到飞书导出菜单项，无需重启浏览器。
5. **Given** 用户通过右键触发"AI 总结本页", **When** 侧边栏尚未打开, **Then** 侧边栏被打开并定位到总结功能、自动开始总结，不丢失结果。
6. **Given** 用户切换界面语言, **When** 再次点右键, **Then** 菜单标题跟随新语言。

---

### User Story 3 - 侧边栏任何页面可见全部能力 (Priority: P1)

作为用户，无论我停在哪个页面，打开侧边栏都应看到同一套可预期的功能列表：顶部是"当前页能做什么"（随页面类型变化的上下文区），下方是"任何页面都能做什么"（常驻的通用工具区），而不是"飞书页看不到剪藏、普通页看不到飞书导出"的二选一。

**Why this priority**: 这是本次重构的主体。功能可见性依赖"用户停在哪"是当前最大的可发现性问题，多个功能（视频下载、截图）因此被藏得很深。

**Independent Test**: 依次在飞书文档页、YouTube 页、B 站视频页、普通网页、浏览器内部页打开侧边栏，核对上下文区内容随页面变化、通用工具区始终存在且分组一致。

**Acceptance Scenarios**:

1. **Given** 用户在飞书文档页打开侧边栏, **When** 查看上下文区, **Then** 看到页面识别条（图标+文档标题+类型）、四个主导出按钮（Markdown/PDF/HTML/附件）、一行"转发布"（小红书卡片、公众号，样式下拉直选）、一项"缓存到本地"。
2. **Given** 用户在飞书文档页, **When** 查看通用工具区, **Then** 剪藏、截图、AI 总结、页面开关分组全部可见（不再隐藏），对飞书页执行剪藏走通用管线。
3. **Given** 用户在 YouTube 视频页, **When** 查看上下文区, **Then** 看到页面识别条、字幕卡；桥接服务就绪时另见视频下载卡。
4. **Given** 用户在其他已支持的视频站点（如 B 站）, **When** 查看上下文区, **Then** 看到页面识别条与视频下载能力入口（桥接服务就绪时）。
5. **Given** 用户在普通网页, **When** 打开侧边栏, **Then** 上下文区只有页面识别条（显示网页标题），剪藏动作在通用区不重复出现。
6. **Given** 用户在浏览器内部页等不可操作页面, **When** 打开侧边栏, **Then** 看到"此页面不可操作"提示条，通用工具区置灰。
7. **Given** 用户点击小红书或公众号旁的样式下拉, **When** 选中一个样式, **Then** 立即执行转换（一次点击选样式，消灭"先展开再选"两段式）。
8. **Given** 用户折叠了某个通用工具分组并关闭侧边栏, **When** 再次打开侧边栏, **Then** 折叠状态保持；首次使用时剪藏组展开、其余折叠。
9. **Given** 用户查看侧边栏顶部与底部, **When** 对照旧版, **Then** 「免费」徽标与配额占位已删除；顶部为状态点+设置+缓存库三个图标；底部页脚为"反馈 · 诊断 · 版本号"小字链接；永久禁用的 Word 占位卡片已删除。
10. **Given** 用户点击顶部缓存库图标, **When** 进入缓存页, **Then** 可查看缓存内容并可返回主页（原「查看缓存」独立卡片不再出现在动作区）。

### User Story 4 - 键盘快捷键直达高频动作 (Priority: P2)

作为高频用户，我希望用键盘快捷键直接触发最常用的动作（打开面板、整页转 Markdown、整页截图），不必碰鼠标。

**Why this priority**: 提效增强项，依赖统一动作分发层（US2 已建立），但缺失它不阻塞核心功能。

**Independent Test**: 在浏览器快捷键设置页确认扩展注册了 4 个命令；逐个按下默认键验证动作触发；验证"总结"命令无默认键但可手动配置。

**Acceptance Scenarios**:

1. **Given** 扩展已安装, **When** 按下"打开面板"快捷键, **Then** 侧边栏打开。
2. **Given** 用户停在普通网页, **When** 按下"整页转 Markdown"快捷键, **Then** 当前页转换并复制，结果反馈遵循角标规则。
3. **Given** 用户停在普通网页, **When** 按下"整页截图"快捷键, **Then** 对当前页执行整页截图（固定 PNG 格式）。
4. **Given** 用户在浏览器快捷键设置页, **When** 查看本扩展, **Then** 可见 4 个命令，其中"AI 总结"默认无键、可自行配置；配置后按键行为与右键"AI 总结本页"一致。

---

### User Story 5 - 后台触发动作的结果可感知 (Priority: P2)

作为用户，我通过右键或快捷键触发动作后（此时侧边栏可能没开），希望能从工具栏图标角标知道成功/失败，失败时能打开侧边栏看到具体原因。

**Why this priority**: 右键/快捷键入口（US2/US4）的体验闭环；无它则后台失败静默丢失。

**Independent Test**: 在侧边栏关闭状态下右键触发截图（一次成功场景、一次构造失败场景），观察角标变化与侧边栏内的失败详情。

**Acceptance Scenarios**:

1. **Given** 侧边栏未打开, **When** 右键触发的动作成功完成, **Then** 工具栏图标出现绿色"✓"角标。
2. **Given** 侧边栏未打开, **When** 动作失败, **Then** 图标出现红色"!"角标；打开侧边栏后能在状态栏看到该标签页任务的失败原因。
3. **Given** 图标带有角标, **When** 用户打开侧边栏或切换到产生角标的标签页, **Then** 角标清除。
4. **Given** 同一标签页先后触发两个任务, **When** 两个任务先后结束, **Then** 角标显示后结束者的状态，但两条任务记录各自保留、均可在侧边栏查看。
5. **Given** 两个不同标签页各有任务, **When** 任务结束, **Then** 任务记录按标签页隔离，互不覆盖。

---

### Edge Cases

- 用户触发动作后立即切换标签页：动作必须继续作用于触发时的那个页面，绝不能"跟着"用户切到新页面（触发瞬间锁定目标页面，全链路不得重新查询"当前活动页"）。
- 触发动作后目标标签页被关闭：任务终止并记录失败原因，不得作用于其他标签页。
- 右键/快捷键触发"AI 总结"时侧边栏正在加载：导航意图必须经持久化传递，侧边栏加载完成后仍能拿到并执行，不因时序竞态丢失。
- AI 总结未配置服务端点时从右键/快捷键触发：侧边栏打开并定位到总结区的配置引导，不静默失败。
- 在浏览器内部页、扩展商店页等受限页面按快捷键或点右键：动作不执行，给出可感知的失败反馈（角标 + 任务记录），不抛静默错误。
- 私有化域授权被移除后：该域的飞书右键菜单项同步消失。
- 视频站点但桥接服务未连接：上下文区不显示下载卡（或显示引导），不出现点击后无响应的死按钮。
- 升级安装（老用户）：原弹窗消失后首次点击图标直接进侧边栏，无需任何迁移操作；折叠态等新偏好按默认值初始化。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 点击工具栏图标必须一步打开侧边栏；原状态弹窗整体删除，不保留任何弹窗入口。
- **FR-002**: 侧边栏顶部必须有桥接服务状态圆点（绿=已连接、灰=未连接、黄=连接中），点击展开详情（服务版本、可复制的会话标识），信息量等价替代原弹窗。
- **FR-003**: 所有右键菜单项必须收敛在一个 LarkSnap 父菜单下，不污染右键一级菜单。
- **FR-004**: 普通网页右键菜单必须包含：整页转 Markdown、选区转 Markdown（仅有选区时）、整页截图（固定 PNG）、AI 总结本页、解除复制限制、打开 LarkSnap 面板。
- **FR-005**: 飞书文档页（官方域及已授权私有化域）右键菜单必须包含：导出为 Markdown、导出为 PDF、导出为 HTML、更多导出方式…（打开侧边栏）。
- **FR-006**: 私有化域授权新增或移除时，右键菜单的域名匹配范围必须随之更新，无需重启浏览器。
- **FR-007**: 界面语言切换后，右键菜单标题必须随之重建为新语言。
- **FR-008**: 必须注册 4 个键盘命令：打开面板、整页转 Markdown、整页截图（以上三个带默认键）、AI 总结（不带默认键，用户可自行配置）。
- **FR-009**: 右键、快捷键、侧边栏按钮三类入口必须收敛到同一个动作分发层；触发瞬间必须捕获目标页面标识与地址并全链路显式传递，分发层之下的任何实现不得自行重新查询"当前活动页"。
- **FR-010**: "AI 总结"从右键/快捷键触发时只发导航意图（打开侧边栏 + 定位到总结功能 + 自动开始）；意图必须经持久化传递，消除侧边栏加载时序竞态；总结不得在无界面承载的情况下于后台直接执行。
- **FR-011**: 后台触发的动作必须以图标角标反馈结果：绿"✓"成功、红"!"失败；失败原因必须写入按标签页隔离的任务记录，可在侧边栏状态栏查看；角标在用户打开侧边栏或切换到该标签页后清除；同一标签页并发任务后者覆盖前者的角标、任务记录各自保留。不引入系统通知。
- **FR-012**: 侧边栏主页必须为单列滚动的分组布局：顶部上下文区（随页面类型变化）+ 通用工具区（常驻）+ 底部固定状态栏 + 页脚。
- **FR-013**: 上下文区必须按页面类型渲染：飞书页（识别条 + 四主导出按钮 + 转发布行 + 缓存到本地）、YouTube 页（识别条 + 字幕卡 + 视频下载卡）、其他视频站点（识别条 + 视频下载入口）、普通页（仅识别条）、受限页（不可操作提示 + 通用区置灰）。页面类型模型必须扩展以区分"视频站点"。
- **FR-014**: 页面识别条必须展示页面图标/类型与标题，让用户确认"即将操作的是哪个页面"。
- **FR-015**: 小红书卡片与公众号转换必须提供样式下拉直选：展开可见样式名与缩略预览，点选样式即执行，全程一次点击选样式。
- **FR-016**: 通用工具区必须在所有页面类型下常驻（受限页置灰），分组为：网页剪藏、截图、AI 总结、页面开关；各分组可折叠，折叠态持久保存；默认剪藏展开、其余折叠。
- **FR-017**: AI 总结分组在未配置服务端点时必须显示一句引导与"去设置"入口；配置后在任何 http 页面可用；YouTube 页自动携带字幕上下文。
- **FR-018**: 侧边栏顶部必须删除「免费」徽标与配额占位，改为状态点、设置、缓存库三个图标；「查看缓存」入口并入缓存库图标，缓存页保持可返回的二级页面。
- **FR-019**: 永久禁用的 Word 导出占位卡片必须删除；「诊断」「反馈」必须移至页脚小字链接，与版本号并列。
- **FR-020**: 现有二级视图（缓存页、小红书预览页）的行为必须保持不变，本次仅重构主页内部组织。
- **FR-021**: 所有新增/调整的界面文案与菜单标题必须提供中英双语。
- **FR-022**: 视频下载入口不进右键菜单（依赖运行时探测，与静态菜单模型不符）。
- **FR-023**: 「写入飞书文档」「采集到多维表格」本次不提供任何 UI 入口，作为独立后续计划。

### Key Entities

- **动作（Action）**: 用户可触发的一次操作（整页转 Markdown、截图、飞书导出、AI 总结导航等）。包含：
  - 动作标识（唯一，枚举固定）
  - 目标页面标识与地址（必填，触发瞬间捕获，不可中途改判）
  - 触发来源（右键 / 快捷键 / 侧边栏按钮之一）
- **页面上下文（PageContext）**: 侧边栏对当前页面的分类结果。包含：
  - 页面类型（飞书 / YouTube / 视频站点 / 普通 / 受限，枚举完备互斥）
  - 页面标题、地址（识别条展示用）
  - 视频站点标识（仅视频站点类型时存在）
- **任务记录（TaskRecord）**: 一次后台触发动作的结果凭证。包含：
  - 所属标签页标识（记录按标签页隔离）
  - 动作标识、结束状态（成功 / 失败）、失败原因（失败时必填）
  - 同一标签页多条记录各自保留；角标只反映最近一条
- **导航意图（NavigationIntent）**: 右键/快捷键触发"AI 总结"时传给侧边栏的指令。包含：
  - 目标功能位置、是否自动开始、目标页面标识与地址
  - 必须持久化存放、被消费后清除（一次性）
- **界面偏好（UIPreference）**: 用户对侧边栏的个性化状态。包含：
  - 各通用分组折叠态（持久，默认剪藏展开、其余折叠）

## Success Criteria *(mandatory)*

- **SC-001**: 从点击工具栏图标到看到可操作功能，点击次数从 2 次降为 1 次。
- **SC-002**: 右键菜单可触达的动作数从 3 个提升到普通页 6 个、飞书页 4 个。
- **SC-003**: 在任意 5 类页面（飞书 / YouTube / 视频站点 / 普通 / 受限）打开侧边栏，通用工具 4 个分组 100% 可见（受限页置灰视为可见）。
- **SC-004**: 小红书/公众号从"想转"到"开始转"的点击次数从 2 次降为 1 次（样式下拉直选）。
- **SC-005**: 触发动作后立即切换标签页的场景下，动作 100% 作用于触发时的页面（0 起错标签页操作）。
- **SC-006**: 侧边栏关闭状态下后台触发的失败，100% 可通过"角标 → 打开侧边栏"路径查到失败原因。
- **SC-007**: 4 个高频动作（开面板、整页 MD、截图、总结）全部具备键盘触达路径（前三者含默认键）。
- **SC-008**: 菜单生成、页面分类、动作分发三块纯逻辑具备单元测试，全部通过。
