# Feature Specification: 已授权域名清单（新建文档免喂链接）

**Feature Branch**: `007-authorized-hosts`
**Created**: 2026-07-16
**Status**: Draft
**Input**: 基于 docs/plans/2026-07-16-已授权域名清单接口.md 生成功能规格：已授权域名清单接口（new-doc 不用每次喂链接）

## Clarifications

以下问题在源计划文档（docs/plans/2026-07-16-已授权域名清单接口.md，已经两轮评审）中已有明确结论，直接采纳：

- Q: 域名清单的数据源是自己另存一份配置，还是复用扩展已有的授权状态？ → A: 复用扩展已有的授权状态，零维护。用户在侧边栏授权/撤销一个域名，清单自动同步，永不失同步，不引入独立配置文件。
- Q: 只有基础域名，够不够定位到一个能新建文档的页面？ → A: 不够。授权记录里存的是基础域，而新建文档需要打开一个真实的租户页面（基础域首页往往是营销站或非目标页面）。所以必须额外补一份"真实租户入口"：优先用授权那一刻用户所在页面的入口，其次用当前已打开的该域名标签页，都没有时明确让用户本次提供一个链接。
- Q: 公有云域名和私有域名在"免喂链接"体验上是否一致？ → A: 不一致，这是已知局限。私有域名有"用户授权"这个时机可以记录真实入口，命中率高；公有云域名是内置的、没有授权动作可捕获，只有用户正开着相应页面时才能补上入口。此局限写入使用文档即可，本期不做增强。
- Q: 要不要做域名别名、默认域名？ → A: 都不做（YAGNI）。一个域名时天然默认；多个域名时强制用户显式指定，支持唯一后缀简写。
- Q: 飞书个人版（公有云下的特殊子域）要不要特殊处理？ → A: 不做特殊标记，仅在使用文档中提示"不保证一次成功"。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 单域名用户新建文档免喂链接 (Priority: P1)

作为一个只使用一个飞书环境（如一个私有化部署域名）的命令行用户，我需要在新建云文档时不再每次手动提供一个该域名下的链接，以便把"写好内容→一条命令建文档"变成零额外输入的操作。

**Why this priority**: 这是本功能的核心闭环——"省链接"体验真正落地的场景，也是私有化用户（最痛的人群）的最常见形态。

**Independent Test**: 在只授权了一个私有域名的环境里，不带任何链接执行新建文档命令，验证文档在该域名下成功创建。

**Acceptance Scenarios**:

1. **Given** 用户已授权且仅授权一个私有域名，且授权时系统记录了真实租户入口, **When** 用户不提供链接执行新建文档, **Then** 文档在该域名的租户下成功创建，全程无需用户补任何链接。
2. **Given** 该域名没有已记录的租户入口，但浏览器正开着一个该域名下的页面, **When** 用户不提供链接执行新建文档, **Then** 系统用已打开页面定位租户入口，文档成功创建。
3. **Given** 该域名既无记录的入口、浏览器也没开着相关页面, **When** 用户不提供链接执行新建文档, **Then** 系统给出明确提示：请本次提供一个该域名下的链接，或先在浏览器打开一个该域名页面——不得静默失败。

### User Story 2 - 查看已授权域名清单 (Priority: P1)

作为命令行用户，我需要一条命令查看当前浏览器扩展已授权的全部域名（内置公共域名 + 我手动授权的私有域名），以便知道新建文档时可以选哪些域名、以及某个域名为什么不可用。

**Why this priority**: 清单是"免喂链接"的前提数据，也是所有错误提示（域名歧义、域名未授权）的用户出口——报错时要能告诉用户"你有哪些可选项"。

**Independent Test**: 执行查看清单命令，输出与扩展侧边栏中实际的授权状态逐项一致。

**Acceptance Scenarios**:

1. **Given** 用户授权过若干私有域名, **When** 执行查看清单命令, **Then** 输出包含全部内置公共域名和已授权私有域名，每项标注来源类型（内置/手动授权）和是否已有可用的租户入口。
2. **Given** 用户在侧边栏新授权了一个私有域名, **When** 再次执行查看清单命令, **Then** 新域名自动出现在清单中，无需任何手工配置。
3. **Given** 用户撤销了某个私有域名的授权, **When** 再次执行查看清单命令, **Then** 该域名从清单中消失，其记录的租户入口也一并清除，无残留。

### User Story 3 - 多域名用户显式选择目标域名 (Priority: P2)

作为同时使用多个飞书环境（公有云 + 多个私有域名）的用户，我需要在新建文档时用一个简短的域名参数指定目标环境，以便在多环境间切换时仍然不用翻找和粘贴链接。

**Why this priority**: 多环境用户是本功能的进阶人群；单域名场景（P1）跑通后，这层只是"选择"逻辑。

**Independent Test**: 在授权了两个以上域名的环境里，分别用完整域名和唯一后缀简写指定目标，验证文档建到正确的域名下。

**Acceptance Scenarios**:

1. **Given** 清单中有多个域名, **When** 用户新建文档且用完整基础域指定目标, **Then** 文档建在指定域名下。
2. **Given** 清单中有多个域名, **When** 用户用一个只能唯一匹配到某个域名的后缀简写指定目标, **Then** 视同完整域名，文档建在匹配的域名下。
3. **Given** 清单中有多个域名, **When** 用户新建文档但没有指定目标域名, **Then** 系统报"域名歧义"错误，并列出全部可选域名让用户选择。
4. **Given** 用户指定的域名不在清单中（或简写匹配到多个）, **When** 执行新建文档, **Then** 系统报"域名未授权"（或歧义）错误，并附上可选清单，提示去浏览器授权或换一个写法。

### User Story 4 - 多浏览器环境按环境查询 (Priority: P3)

作为在同一台电脑上用多个浏览器用户身份（各自装有扩展、授权域名不同）的用户，我需要在查询清单和新建文档时指定用哪个浏览器身份，以便清单和建档行为对应到正确的登录态。

**Why this priority**: 多浏览器身份是小众场景，但不处理会导致"清单混淆"这类难排查的错误。

**Independent Test**: 两个浏览器身份各授权不同域名，分别指定身份查询，验证各自返回自己的清单。

**Acceptance Scenarios**:

1. **Given** 只有一个浏览器身份在线, **When** 不指定身份查询清单或新建文档, **Then** 自动使用该身份，行为正常。
2. **Given** 多个浏览器身份同时在线, **When** 不指定身份执行, **Then** 系统报"身份歧义"错误，提示指定身份参数。
3. **Given** 多个浏览器身份同时在线, **When** 指定其中一个身份查询清单, **Then** 返回的是该身份扩展的授权域名，不混入其他身份的。

### Edge Cases

- **旧版本扩展**：用户浏览器里装的扩展还是旧版本、不支持提供域名清单时，系统必须明确提示"更新扩展，或按旧用法提供一个链接"，不得静默失败或给出误导性错误。
- **旧用法完全兼容**：用户仍按老习惯直接提供链接新建文档时，行为与现在完全一致，不受本功能影响。
- **其他编辑操作不放宽**：追加、插入、替换等需要定位到具体文档的操作，仍然必须提供文档链接；对这些操作传"目标域名"参数时系统明确拒绝，避免用户误以为也能免链接。
- **公有云域名入口缺失**：内置公共域名通常没有已记录的租户入口，用户没开着相应页面时会走"请本次提供链接"的降级提示——这是已知局限，需写入使用文档。
- **同一域名多租户**：同一个基础域下用户可能属于多个租户，系统只记录最近一次授权时的入口；建错租户时用户可用旧用法（直接提供目标租户链接）纠正。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供"已授权域名清单"查询能力，清单包含内置公共域名和用户手动授权的私有域名两类，并逐项标注来源类型。
- **FR-002**: 清单必须直接来源于扩展的实际授权状态：新授权的域名自动出现，撤销授权的域名自动消失；禁止引入需要人工维护的独立域名配置。
- **FR-003**: 用户授权一个私有域名时，系统必须同时记录该时刻用户所在页面的真实租户入口；撤销该域名授权时，此记录必须一并清除，不得残留。
- **FR-004**: 清单中每个域名必须解析"真实租户入口"，解析优先级为：授权时的记录 → 当前浏览器中已打开的该域名页面 → 无（明确标注为空）。
- **FR-005**: 新建文档命令必须支持不提供链接：清单中只有一个域名时直接使用它；有多个域名时要求用户显式指定目标域名，未指定则报"域名歧义"并附可选清单。
- **FR-006**: 目标域名参数必须支持完整基础域和唯一后缀简写两种写法；匹配不到或匹配到多个时报错并附可选清单。
- **FR-007**: 免链接新建文档必须使用该域名的"真实租户入口"定位建档页面；入口为空时必须提示用户"本次提供一个链接，或先打开一个该域名页面"，禁止静默失败，也禁止拿基础域硬拼页面地址。
- **FR-008**: 存在多个浏览器身份时，清单查询与免链接新建文档必须支持身份参数；多身份在线且未指定时报"身份歧义"错误。
- **FR-009**: 提供链接的旧用法必须完全向后兼容，行为不变。
- **FR-010**: 除新建文档外的编辑操作维持"必须提供文档链接"的要求；这些操作携带目标域名参数时必须被明确拒绝。
- **FR-011**: 扩展版本过旧、不具备提供清单能力时，系统必须返回明确的"扩展需更新"错误，并提示两条出路：更新扩展，或按旧用法提供链接。
- **FR-012**: 所有失败路径（域名歧义、域名未授权、入口缺失、身份歧义、扩展过旧、扩展未连接）都必须有可区分的错误标识和面向用户的下一步指引。

### Key Entities

- **授权域名条目**：清单中的一项，代表一个用户可在其下操作的域名。包含：
  - 基础域（必填，如 `a.example.com`；是域名本身，不是通配模式）
  - 来源类型（必填，二选一：内置公共域名 / 用户手动授权的私有域名）
  - 真实租户入口（可选，该域名下一个能实际承载建档操作的页面来源地址；为空表示当前无法免链接建档）
- **租户入口记录**：随授权动作产生、随撤销动作删除的附属记录。以基础域为键，值为授权那一刻用户所在页面的入口地址；同一基础域只保留一条（最近一次）。它是对现有授权记录的丰富，不是第二份独立配置。
- **浏览器身份**：一个装有扩展并保持连接的浏览器用户环境。清单按身份隔离——每个身份有自己的授权域名和入口记录，查询结果不跨身份合并。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 已授权私有域名的用户新建文档，所需手动提供的链接数从每次 1 个降为 0 个（授权动作之后的所有建档均免链接）。
- **SC-002**: 多域名用户新建文档，除内容和标题外只需 1 个域名参数（且支持简写），不需要粘贴任何 URL。
- **SC-003**: 授权状态与清单同步率 100%：新授权后立即可见，撤销后立即消失且入口记录零残留（手测撤销场景逐项验证）。
- **SC-004**: 全部失败路径（至少覆盖：域名歧义、域名未授权、入口缺失、身份歧义、扩展过旧、旧用法回归）在手测中均给出明确、可行动的错误提示，静默失败 0 例。
- **SC-005**: 使用旧用法（提供链接）的既有流程回归通过率 100%，无任何行为变化。
