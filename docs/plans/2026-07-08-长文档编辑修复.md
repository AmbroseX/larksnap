# 长文档编辑修复（append 定位 / 回读校验收尾）

> 背景：2026-07-08 实验 4（见 2026-07-07-飞书文档编辑.md）确认飞书长文档虚拟化渲染下
> DOM 最多只挂约 238 块。同日已修好 `fetchClientVars` 翻页（mode=1 + cursor + limit，
> 771ffb1），服务端全量块数据可得。
>
> 2026-07-08 评审对照代码后翻案：原计划的三处修复里，list-blocks 已经是服务端读取
> （从来不看 DOM），服务端回读终验也已经存在；真正要修的只有 append 定位一处，
> 外加回读链路的两个收尾。本文档是评审后的修订版。

## 一、现状盘点（对照代码，别重做已有的东西）

| 原计划条目 | 代码现状 | 结论 |
|---|---|---|
| list-blocks 改走 client_vars | 一直就走 `readTree` → `fetchClientVars`（editor.ts），从不枚举 DOM；之前"只报 238 块"是翻页缺失所致，771ffb1 已修 | **不用做**，跑一次 424 块文档确认即可 |
| 回读校验改走服务端 | `verify()` 已每轮重拉全量 client_vars，查首末行指纹 / 旧块 ID 消失，且有 2s→3s→自适应三轮重试 | **已存在**，剩两个收尾（见 2.2） |
| append 定位修复 | locate 的滚动扫描明确排除 append（editor-inject.ts `p.mode !== 'append'`），目标块不在 DOM 就退回"DOM 末块"，长文档插进中间 | **真 bug，主修这个** |

一句话总思路不变：**读靠接口，写靠跳转，验靠服务端**——只是前两条已经落地，本轮把第三条补上，并让"验"真正成为最终裁决。

## 二、两处修改

### 2.1 append 定位：先滚到文档真末尾再定位（主修）

**现状**：editor-inject.ts 的 locate 步骤里，滚动扫描只给 after/replace/delete 用，
append 模式被排除；长文档下服务端给的"真末块 ID"不在 DOM 里，代码退回"取 DOM 里
最后一个块"，那是渲染窗口的末块，内容插进文档中间。

**修法**：复用旁边已有的滚动机器，不引入新手段：

1. append 且目标块不在 DOM 时，把 scroller 滚到 `scrollHeight`，等一拍让虚拟化补渲染；
2. scrollHeight 会随渲染增长，循环"滚到底 → 等待"直到高度稳定（和现有 settle 同款判稳）；
3. 稳定后再 `findBlockNode(blockId)` 找服务端给的真末块——此时它就在视口里，
   走现有"点击末块文本末端 → 回车 → 粘贴"，后面全部不变。

不用 Cmd/Ctrl+End 快捷键方案：它依赖飞书快捷键处理器和 CDP 焦点仿真状态，是新的
不确定性，还得先做实验验证；滚动方案零实验、改动只在 locate 一处（KISS / YAGNI）。

### 2.2 回读链路两个收尾

**收尾 A：DOM 中间校验降级为快速通道，失败不短路。**

现状：cdpInject 里的 DOM 指纹校验（editor-inject.ts 的 verify 步骤）是硬门槛——
超时返回失败后 runEditJob 直接报 `inject_failed` 结束，根本走不到服务端终验，
哪怕内容其实已经落地。长文档下 DOM 判据本来就不可靠（虚拟化），不该由它一票否决。

修法：DOM 校验通过 → 照旧走服务端确认（快速通道）；DOM 校验超时 → 不再报错返回，
带着"DOM 未确认"标记继续落到服务端终验，由服务端数据下最终结论。超时仍未在服务端
见到指纹才报 `save_unconfirmed`。

顺带纠偏：delete/replace 的"旧块节点消失"DOM 判据被虚拟化骗过，后果只是误报失败
（服务端 verify 已用 block_map 兜底判真删），不是正确性隐患，不会脏数据——所以这里
只做降级，不做重写。

**收尾 B：append 的服务端判据从"包含"升级为"位置"。**

现状：`verify()` 对 append 只查"首末行出现在服务端全文里"——包含性检查分不出
"插在末尾"还是"插在中间"，2.1 修没修好它都报成功，验收只能靠人工导出比对。

修法：append 判据升级为**服务端顺序上最后一个有文本的块 == 源内容末行**（剥
Markdown 语法、空白归一后比对）。分片粘贴会在尾部多留空段落，所以取"最后一个
非空文本块"而不是字面末块。其他模式判据不动。

**成本提醒**：`verify()` 每跑一次是整份 client_vars 全量翻页（424 块 = 首屏加多次
续页，外加 skip_blocks 补拉）。现有的三轮自适应重试已经够用，不改成密集轮询。

## 四、实施顺序与验收

顺序：**第 0 步重验现状 → 2.1（真 bug）→ 2.2 两个收尾**。

第 0 步：在修好翻页的读管线上重跑一遍原故障场景，确立真实基线——
list-blocks 数块、32KB append。很可能 save_unconfirmed 误报已被翻页修复顺带治好，
重跑结果决定 2.2 收尾 A 的优先级。

验收（424 块长文档 + 短文档回归，各命令都跑）：

1. list-blocks 报满 424 块，首末块 summary 与源一致（第 0 步即可确认，属回归项）；
2. 32KB append 返回成功（不再 save_unconfirmed），服务端导出全文比对零缺失；
3. 长文档上 append 小内容，**回读校验自己就能确认落在文档末尾**（收尾 B 的位置判据），
   再用导出人工复核一次；
4. 长文档中部块的 replace-block / delete-block，服务端确认旧块消失、新内容在位；
5. 人为制造 DOM 校验超时（或用长文档触发），确认不再报 inject_failed 短路，
   而是走到服务端终验给出正确结论（收尾 A）；
6. 短文档回归：六个子命令全过，速度无明显退化。

## 六、边界与不做的事

- 不做 insert-after 重写：locate 的滚动扫描已覆盖 after 模式，长文档下按块 ID
  定位本来就能用；标题文本匹配走服务端树，也不受虚拟化影响；
- 不做 Cmd+End 快捷键跳转（见 2.1，滚动方案零实验零新依赖）;
- 不把 verify 改成密集轮询：全量翻页成本在那，三轮自适应重试够用；
- 不动协同协议：仍然只做"像用户一样"的输入，服务端数据只读；
- settle / WS 静默等待等现有机制全部保留。
