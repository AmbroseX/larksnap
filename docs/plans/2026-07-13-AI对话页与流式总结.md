# AI 对话页与流式总结升级方案

日期：2026-07-13
状态：阶段 A/B/C 代码已完成（typecheck + 单测 + 构建通过），待真机验证，未提交
前置：006 交互入口与侧边栏重构（代码已完成、待真机验证）

## 一、背景与问题

用户反馈四点，全部指向 AI 总结从"一张卡片"升级为"一个对话页"：

1. **侧边栏应有两个页面**：常规工具页 + AI 对话页，顶部左右切换（Monica 的多 Tab 形态）；
2. **总结后要能继续对话**：现在 SummaryView 只出一次结果，无法追问；
3. **分块总结又慢又贵**：现状每 1000 字切一块、串行"总结→精炼"（`summarize/splitter.ts` + refine 循环），一篇文章十几次请求；这是 004 期为兼容小上下文端点 + 失败保部分结果做的保守设计，对现代模型完全没必要；
4. **结果无渲染、无流式**：`chatComplete` 是 `stream:false` 一次性返回（`llm.ts:67`），UI 用 `<textarea>` 裸展示 Markdown 源码。

## 二、方案总览

```
┌─────────────────────────────┐
│ LarkSnap  ●  [工具] [AI对话] ⚙ 🗂 │ ← header 加双页 Tab（左右切换）
├─────────────────────────────┤
│  工具页 = 006 的 home        │
│  （上下文区 + 通用工具区）      │
├─────────────────────────────┤
│  AI 对话页：                  │
│  ┌ 会话条：新会话 · 历史会话 ▾ ┐│
│  │ [assistant] 总结内容(md渲染)││ ← 流式增量渲染
│  │ [user] 追问...             ││
│  │ [assistant] 回答(md渲染)   ││
│  └ 输入框 [发送]              ┘│
└─────────────────────────────┘
```

### 2.1 双页形态（左右切换）

- `SidePanel` 的 view 状态机扩展为 `home | chat | cache | xhsPreview`；header 标题旁放两个 Tab（工具 / AI 对话），只在 `home ⇄ chat` 间切换；cache、xhsPreview 仍是带返回的二级页。
- 记住上次停留页：存**独立键** `larksnap:last-view`，不并入 `larksnap:ui-prefs`——ToolGroups 保存折叠态是整个对象覆盖写（只带 collapsedGroups），共用一个 key 会把 lastView 冲掉（评审 P2）。
- 「AI 总结」的各入口全部改为**跳到对话页**：右键/快捷键的导航意图（006 已有 `NavigationIntent`）消费后切到 chat 页新建会话自动开跑；工具页通用区的"AI 总结"组简化为一个入口按钮（未配置端点时仍显示引导+去设置）。

### 2.2 总结策略：按 token 预算决定直塞或切块（评审阻断项 1）

固定字符阈值不安全——中文接近一字一 token，60k 字符可能远超 16k/32k 上下文的端点。改为按 token 预算：

- 新增纯函数 `estimateTokens(text)`：CJK 字符计 1 token，其余按 4 字符 ≈ 1 token 估算（故意偏高，宁可提前切块也不溢出）；
- 预算默认 24,000 tokens（先写死常量，后续有需要再进设置页）；首轮 messages 整体估算 ≤ 预算 → 整篇直塞；超过 → 切块 refine，块大小按预算折算（而非固定字符数），overlap 500 字符；
- **溢出自动降级**：端点返回上下文溢出类错误（HTTP 4xx 且报文含 context / length / token 关键词）时，自动降级为切块路径重试一次，不直接报死；
- **追问历史裁剪**：每次发送前对整个 messages 数组估算，超预算按顺序执行——① 把首轮 user 的页面全文替换为已生成的总结（前面加一句"原文已省略"）；② 仍超则从最旧的追问轮开始成对丢弃；system 和最近一轮永不裁剪；
- `splitter.ts` 不动，只改调用处的判定逻辑；估算与裁剪都是纯函数，配单测。

### 2.3 生成协议：prepare（普通消息）→ Port 流式（评审阻断项 2）

前置状态（首次使用的 needsAck 确认、私有化域名的 needsPermission 授权）没法塞进流式协议——`chrome.permissions.request` 必须在侧边栏的用户手势调用栈里发起。所以拆两段：

1. **prepare（现有 sendMessage 通道）**：新消息 `SUMMARIZE_PREPARE { target }`，SW 检查端点配置、权限、取材可行性，返回 `ok | needsAck | needsPermission(origin) | error(kind)`；UI 据此弹确认 / 在手势内调 `permissions.request`，全部就绪后才进流式。
2. **流式（长连接）**：侧边栏 `chrome.runtime.connect({ name: 'larksnap-chat' })`。每次生成分配自增 **requestId**（评审 P2，防串流污染）：
   - UI → SW：`{ type:'start', requestId, sessionId, messages }`、`{ type:'stop', requestId }`
   - SW → UI：`{ type:'delta', requestId, text }` / `{ type:'done', requestId, full }` / `{ type:'error', requestId, kind, message }`
   - UI 只接受等于当前 requestId 的事件，迟到的旧流事件直接丢弃；**每个会话同时只允许一个进行中的生成**，发新 start 前必须先 stop 旧的。
3. Port `onDisconnect`（关侧边栏）→ SW 对该 Port 进行中的请求 `AbortController.abort()`，不白烧 token；用户 stop、断连、120s 超时三者合并到同一个 signal（超时用 `setTimeout` + 同一个 controller 实现，不再单独起一套）。

`llm.ts` 新增 `chatCompleteStream(ai, messages, onDelta, signal)`：`stream:true`，用 `res.body` 的 reader 解析 SSE。解析拆为可单测的纯函数，明确要求（评审补充项）：

- 跨 chunk 的**尾部缓冲**：一行 `data:` 可能被 reader 切成两半，未完整的尾巴留到下一轮拼接；
- `TextDecoder` 用 `{ stream: true }` 增量解码，避免多字节字符被切坏；
- 行分隔兼容 `\n` 与 `\r\n`；`data: [DONE]` 结束；每行取 `choices[0].delta.content`；
- 错误分类沿用 `LlmError`。

现有 `chatComplete`（非流式）保留给切块 refine 的中间块用；最后一块走流式吐给 UI。

### 2.4 继续对话

- 会话消息结构：system（角色设定 + 目标语言）+ user（页面全文/字幕，仅第一轮）+ assistant（总结）+ 后续 user/assistant 轮次；追问时发送整个 messages 数组（内容只发往用户自配 baseUrl，宪法 V 不变），发送前先过 2.2 的 token 预算裁剪。
- 输入框支持 Enter 发送 / Shift+Enter 换行；生成中可"停止"。

### 2.5 会话管理

- `ChatSession { id, title(取页面标题), target{tabId,url}, messages, createdAt, updatedAt }`；
- 存 `chrome.storage.session`（键 `larksnap:chat:sessions`，已拍板），**关开侧边栏不丢，浏览器重启清空**；
- **配额按字节不按条数**（评审 P2）：落盘前序列化，总字节 > 1.5MB 就从最旧会话开始裁，直到装得下；20 个的条数上限保留作次级约束；`storage.session.set` 捕获配额错误，失败丢最旧再试一次，仍失败只保留当前会话；
- **只在终态落盘**：done / error / 用户 stop 时写一次，流式过程中不碰 storage；stop 保存已生成的部分回答并在消息上标 `stopped: true`（UI 显示"已停止"角注），断连（侧边栏被关）来不及落盘的部分直接丢弃；
- 会话条：「新会话」按钮 + 历史会话下拉（标题 + 时间），点选载入即可接着聊。

### 2.6 Markdown 渲染

- 用已有依赖 `marked` 渲染 + `highlight.js` 代码高亮，**运行时不引入新依赖**；
- sanitize 升级为 **DOM 遍历白名单式**，不用正则（评审阻断项 3——正则剥 `on*`/`javascript:` 挡不住实体编码、大小写空白变体、SVG/MathML 向量）：
  - `DOMParser` 解析 marked 输出 → 递归遍历，**标签白名单**（p、h1-h6、ul/ol/li、blockquote、pre/code、table/thead/tbody/tr/th/td、a、strong/em/del、hr/br），白名单外的元素连同子树整体丢弃（script/style/iframe/svg/math/form 等全灭）；
  - **属性按标签白名单**：a 只留 href，code 只留 `class="language-*"`，其余属性一律剥掉（`on*` 自然不存在）；
  - href 校验用 `new URL()` 解析后判 protocol 只许 http/https——DOMParser 拿到的属性值已完成实体解码，URL 对象又吞掉大小写与空白变体，绕过面收敛到最小；渲染出的 a 统一 `target=_blank rel=noopener noreferrer`；
  - **img 不渲染**，降级为 `[图: alt]` 文本——远程图片会向第三方泄漏用户在看什么（评审提到的隐私向量），总结场景几乎不产图，不值得开这个口子；
  - 配**恶意载荷单测集**：`onerror`、`JaVaScRiPt:` / 实体编码 / 制表符空白变体、`<svg onload>`、`data:` URL、嵌套 form、残缺标签等（sanitize 单测需要 DOM，vitest 加 jsdom 环境跑，jsdom 仅 devDependency 不进产物）；
- 流式期间对"未闭合的代码块/表格"做容错：每次 delta 后全文重渲（marked 对残缺 md 也能出稳定结果），但**渲染要节流**（评审补充项）：约 150ms 合并一次，流式中间态跳过 highlight.js，done 后终渲一次再补代码高亮——否则每个 token 都全文解析+高亮，长回答会二次方地卡。

## 六、评审结论与拍板（2026-07-13）

外部评审（review.md）的 3 个阻断项已全部吸收进上文：

1. token 预算切块 + 溢出降级 + 历史裁剪 → 2.2；
2. prepare→流式两段协议，needsAck / needsPermission 由 prepare 承载，授权保持在手势内 → 2.3；
3. sanitizer 从正则改为 DOM 遍历白名单 + 恶意载荷测试集 → 2.6。

P2 项同样并入：流事件带 requestId（2.3）、lastView 用独立 storage key（2.1）、会话按字节配额裁剪且只在终态落盘（2.5）。SSE 尾部缓冲/CRLF、Abort+超时合并、渲染节流等补充建议也已写进对应小节。

原待确认三项按评审建议定案：

1. 会话持久化：**storage.session**；
2. 超长兜底：**保留切块 refine**，判定从固定字符数改为 token 预算；
3. **needsAck 沿用**现有交互，载体改为 prepare 阶段的返回值。

唯一新增的破例：sanitize 单测需要 DOM 环境，计划给 vitest 加 **jsdom 作 devDependency**（只进测试，不进扩展产物，不触碰"运行时无新依赖"约束）。若不接受，则该测试集降级为手测载荷清单。

## 七、实施偏差记录（2026-07-13 编码完成后）

1. **summarizePage（非流式一次性总结）保留未删**：dispatch 分发表的 panel 路由与 12 个单测仍引用它，且它同步升级了 token 预算 + 溢出降级逻辑；UI 已不再调用（SUMMARIZE_PAGE 消息成为无调用方的后端能力），后续确认无用时再删。
2. **prepare 含取材**：计划里 prepare 只做"配置/权限/取材可行性检查"，实做把取材结果直接缓存在 SW 内存（sourceId，5 分钟过期）——避免 start 时二次取材，needsPermission 也自然在 prepare 阶段暴露。
3. **会话下拉、意图跳转在阶段 B 一并完成**（原计划排在 C），C 阶段实际只剩收尾清理。
4. **首轮统计埋点迁移**：原 SUMMARIZE_PAGE 处理器里的 summarize 统计（枚举/数值）移到 chat-port 首轮生成的终态处，多报一个 `chat:true` 维度。
5. **已知边界**：进入「缓存库」二级页会卸载对话组件 → Port 断开 → SW 中止进行中的生成（已生成部分会作为 stopped 消息落盘）；home ⇄ chat 切换不受影响（保持挂载只切显隐）。
