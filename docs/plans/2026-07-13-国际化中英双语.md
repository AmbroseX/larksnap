# 国际化方案：中英双语 + 设置内切换

日期：2026-07-13
状态：已实施完成（见第七节实施纪要），待浏览器手动验收

## 一、目标与边界

**要做的：**
1. 支持中文、英文两种界面语言，在设置页里切换，立即生效（不用重启浏览器）。
2. 默认「跟随浏览器」：浏览器是中文环境显示中文，其他环境显示英文。
3. 字典结构留好扩展口子，以后加日语、韩语只需新增一个字典文件。

**不做的（YAGNI）：**
1. 不做日期、数字的本地化格式（当前 UI 里几乎没有这类展示）。
2. 不翻译代码注释、console 日志、桥接 daemon 给 CLI 的输出（`skills/larksnap-fetch/`）——这些不是最终用户可见的界面。
3. 不引入 i18next 等第三方库——总共几百条文案，自己写一个 50 行以内的 `t()` 就够了，还省包体积。

## 二、现状盘点

所有文案都是硬编码中文，用户可见的分布在四类地方：

| 位置 | 文件 | 估算文案量 | 说明 |
|---|---|---|---|
| React 界面 | `src/options/Options.tsx`、`src/popup/Popup.tsx`、`src/sidepanel/*.tsx`、`src/sidepanel/actions.ts` | 约 200 条 | 主体工作量 |
| content script | `src/content/webcopy/toast.ts`、`auto-copy.ts` | 约 6 条 | 页面内 toast 提示 |
| background | `src/background/` 各文件里 throw/返回给 UI 的错误文案、进度文案（progress.ts 推送） | 约 80~100 条 | 只翻译会流到 UI 的字符串，纯日志不动 |
| 右键菜单 | `src/background/webcopy.ts` 的 `registerMenus()` | 3 条 | `chrome.contextMenus.create` 的 title；切语言时必须重建菜单才能生效，见 4.5② |
| manifest | `manifest.json` 的 name / description / default_title | 3 条 | 走 Chrome 官方 `_locales` 机制 |

## 三、技术选型：为什么不用 chrome.i18n 做界面文案

Chrome 自带 `chrome.i18n.getMessage()` + `_locales/` 机制，但它有个硬伤：**语言只跟随浏览器 UI 语言，运行时无法切换**。用户要求"设置里切换语言"，所以：

- **界面文案**：自己写一个轻量 i18n 模块（`src/shared/i18n/`），语言存在扩展配置里，切换即生效。
- **manifest 文案**（扩展名、描述、图标悬停标题）：这几条只能走 `_locales` 机制（manifest 不支持运行时改），跟随浏览器语言。两套机制各管各的，互不冲突。

## 四、方案设计

### 4.1 目录结构

```
src/shared/i18n/
├── index.ts      # t() 函数、语言解析、useI18n hook 之外的纯逻辑
├── useI18n.ts    # React hook（监听 storage 变化，切换即重渲染）
├── zh.ts         # 中文字典（基准，类型来源）
└── en.ts         # 英文字典（类型约束为「与 zh 结构完全一致」）
_locales/
├── zh_CN/messages.json   # manifest 用：扩展名、描述
└── en/messages.json
```

### 4.2 字典与类型安全

以中文字典为基准推导类型，英文字典必须逐 key 对齐，**漏译直接编译报错**：

```ts
// zh.ts
export const zh = {
  options: {
    title: '飞书文档导出助手 · 设置',
    imageMode: {
      label: '导出 Markdown 时的图片处理',
      download: '下载图片到本地（随 zip 打包）',
      link: '保留在线链接',
    },
    // ...
  },
  sidepanel: { /* ... */ },
  popup: { /* ... */ },
  webcopy: { /* ... */ },
  errors: { /* background 会流到 UI 的错误 */ },
  progress: { /* 进度推送文案 */ },
} as const;

// index.ts
type DeepStringify<T> = { [K in keyof T]: T[K] extends string ? string : DeepStringify<T[K]> };
export type Dict = DeepStringify<typeof zh>;

// en.ts
export const en: Dict = { /* 结构少一个 key 就报错 */ };
```

类型安全要覆盖两端：

1. **字典完整性**：`en: Dict` 保证英文字典不缺 key（上面已做到）。
2. **调用方 key 合法性**：`t()` 的 key 参数不能是裸 `string`，否则 `t('options.imageMode.lable')` 这种拼写错误能通过编译。从 `typeof zh` 递归推导出点分路径联合类型：

```ts
type DotPath<T, P extends string = ''> = {
  [K in keyof T & string]: T[K] extends string
    ? `${P}${K}`
    : DotPath<T[K], `${P}${K}.`>;
}[keyof T & string];

export type TranslationKey = DotPath<typeof zh>;
// 得到 'options.title' | 'options.imageMode.label' | ... 的联合类型
```

写错 key 直接编译报错。

### 4.3 t() 函数与占位符

```ts
// 用法：t('options.imageMode.label')
// 带参数：t('progress.exporting', { name: '文档标题' })
// 字典里写：exporting: '正在导出 {name}…'
export function t(key: TranslationKey, params?: Record<string, string | number>): string;
```

- key 用点分路径查找，查不到时回退中文字典，再查不到就原样返回 key（不崩）。
- 占位符用 `{name}` 简单替换，不做复数等复杂规则（YAGNI）。

### 4.4 语言设置的存储与解析

`ExtensionConfig` 加一个字段（`src/shared/types.ts` + `constants.ts` 的 `DEFAULT_CONFIG`）：

```ts
language: 'auto' | 'zh' | 'en';   // 默认 'auto'
```

解析规则（`resolveLanguage`）：

```
'zh' / 'en'  → 直接用
'auto'       → chrome.i18n.getUILanguage() 以 zh 开头 → 'zh'，否则 → 'en'
```

存量用户配置里没有这个字段，`getConfig()` 已有 DEFAULT_CONFIG 兜底合并，自动落到 'auto'，中文浏览器用户无感知。

### 4.5 三种运行环境怎么接入

扩展有三类互相隔离的运行环境，接法各不相同：

**① React 页面（popup / options / sidepanel）— `useI18n` hook**

```tsx
function Options() {
  const { t, lang } = useI18n();
  return <h1>{t('options.title')}</h1>;
}
```

hook 内部：
- 挂载时读 `getConfig()` 拿语言；
- 监听 `chrome.storage.onChanged`，配置里的 language 变了就 setState 重渲染。

效果：在设置页切语言，**已经打开的侧边栏、popup 立即跟着变**，不用刷新。

**② background（Service Worker）— 单例 initPromise + 同步 t()**

SW 里到处是同步抛错的地方，不能每次都 `await getConfig()`。但 SW 冷启动时会立即执行桥接（`startBridge()`）、右键菜单（`setupWebcopy()`）、统计等逻辑，用户文案的入口不止 runtime 消息一个，所以初始化必须是**单例 Promise，谁用谁等**：

```ts
// i18n/index.ts 里额外导出
let currentLang: 'zh' | 'en' = 'zh';
let initPromise: Promise<void> | null = null;

/** 幂等：首次调用读配置 + 注册 onChanged，后续调用复用同一个 Promise */
export function ensureI18n(): Promise<void> {
  initPromise ??= (async () => {
    currentLang = resolveLanguage((await getConfig()).language);
    chrome.storage.onChanged.addListener(/* language 变了更新 currentLang */);
  })();
  return initPromise;
}
```

所有会产生用户文案的 async 入口开头都 `await ensureI18n()`，一个不漏：

- `src/background/index.ts` 的 runtime 消息路由入口；
- `webcopy.ts` 的 `chrome.contextMenus.onClicked` 回调；
- `bridge.ts` 收到 daemon 远程任务的处理入口；
- `registerMenus()` 改为 async，创建菜单前先 `await ensureI18n()`。

这样不存在"冷启动窗口拿到错误语言"——同步 `t()` 只在这些入口之后被调用，而入口已保证 currentLang 就绪。

**右键菜单的"切换即生效"**：菜单标题是创建时固化的，语言变更不会自动刷新。在 i18n 的 onChanged 回调里暴露一个订阅点（`onLanguageChanged(cb)`），`webcopy.ts` 订阅后重新执行 `registerMenus()`（内部本来就是 `removeAll` + `create`，天然幂等）。

**③ content script（webcopy toast）— 注入时随消息带入或直读 storage**

content script 也能访问 `chrome.storage`，与 ① 同样方式读配置即可；toast 文案只有几条，也监听 onChanged 保持同步。

### 4.6 background 错误文案的处理原则

background 里的中文字符串分三类，只动第一类：

1. **会流到 UI 的**（`sendResponse({ error: '...' })`、progress 推送、throw 后被 actions.ts 捕获展示的）→ 收进字典 `errors.*` / `progress.*`。
2. **纯 console 日志** → 保持中文不动。
3. **代码注释** → 保持中文不动。

排查方法：从 `src/sidepanel/actions.ts` 和 `progress.ts` 反向追溯哪些字符串最终被渲染，逐条收编，而不是把 background 所有中文一锅端。

### 4.7 manifest 国际化（跟随浏览器语言）

```jsonc
// manifest.json
"name": "__MSG_appName__",
"description": "__MSG_appDesc__",
"default_locale": "zh_CN",
"action": { "default_title": "__MSG_appName__" }
```

`_locales/zh_CN/messages.json` 和 `_locales/en/messages.json` 各放 3 条。

**明确任务**：`_locales/` 目前不会进 dist——`vite.config.ts` 的 post-build 插件只拷贝了 `icons/*.png`。需在同一处（拷贝图标的代码旁）加一段递归复制 `_locales/` 目录到 dist（`cpSync(src, dest, { recursive: true })` 一行即可），并在构建后核对 dist 里目录存在。

### 4.8 设置页新增语言选择

Options.tsx 最顶部加一个下拉框：

```
界面语言 / Language
├── 跟随浏览器 / Auto
├── 中文
└── English
```

这一项的选项文字本身用「中英并排」写死，保证任何语言状态下用户都能认出来。

### 5.1 验证方案

**自动化（第 1 批落地，之后每批都跑）：**

- `npm run typecheck`：en 字典漏 key、调用方写错 key 都会编译报错。
- vitest 单测（纯函数，不依赖浏览器）：
  - `resolveLanguage`：'zh' / 'en' / 'auto'（zh-CN、zh-TW、en-US、ja 等 UI 语言）各分支；
  - `t()`：正常查找、缺失 key 回退中文、中英都缺回退 key 本身；
  - 占位符：单个、同名多次出现、参数缺失时不崩；
  - `ensureI18n`：并发调用只初始化一次（单例语义）。

**手动验收矩阵（第 3 批完成后过一遍）：**

| 场景 | 验收点 |
|---|---|
| auto + 中文浏览器 / auto + 英文浏览器 | 界面语言分别为中/英 |
| 设置页切 zh ↔ en | 已打开的侧边栏、popup 立即变，不用刷新 |
| 切语言后右键菜单 | 三条菜单标题已重建为新语言 |
| SW 冷启动（扩展重载后直接点右键菜单/发桥接任务） | 文案语言正确，无中文兜底闪现 |
| 导出失败场景（如未登录） | 错误提示随语言切换 |

## 六、风险与注意点

1. **文案收编是体力活，最容易漏**：验收时用正则 `[一-龥]` 扫 `src/` 下的 `.tsx` 和会流到 UI 的字符串，剩下的应只有注释和 console 日志。
2. **英文文案长度**：英文普遍比中文长 30%~50%，sidepanel 按钮较密集，翻译时控制用词，必要时微调 CSS（`white-space`、`min-width`）。
3. **SW 冷启动语言窗口**：见 4.5②，单例 `ensureI18n()`，所有产生用户文案的入口（消息路由、右键菜单回调、桥接任务）都先 await 它。
4. **存量用户无感知**：language 默认 'auto'，中文浏览器用户升级后界面不变。
5. **后续扩展第三种语言**：新增 `ja.ts`（类型约束 `Dict`）→ resolveLanguage 加一个分支 → 设置下拉加一项，三步完成，不动任何业务代码。

2. **两处变量名遮蔽**：`transcript.ts`、`xhs/build-nodes.ts` 里已有局部变量 `t`，i18n 导入改用 `import { t as i18n }`。
3. **JSX 内嵌元素的占位符**：`popup.connectHint` 的 `<code>` 和 `summary.ackIntro` 的 `<strong>`，用 `t(key).split('{占位符}')` 切开渲染，保留样式。
4. **保持中文不译的（有意为之）**：
   - 注入页面主世界执行的函数内字符串（`sheet.ts` 读表诊断、webcopy 适配器的 `note`）——序列化后拿不到 t()；
   - `feishu-proxy.ts` 的 '返回 JSON 非二进制'/'空响应'（内部重试判断）；
   - bridge/editor 面向 CLI 的输出、console 日志、代码注释；
   - 导出 Markdown 里的 HTML 注释（`<!-- 暂不支持的块类型 -->` 等，渲染后不可见）；
   - AI 配置的 targetLang 默认值 '中文'（是数据不是界面文案，用户可改）。
5. **登录态识别正则兼容**：`bridge.ts` 靠 `/登录|not.?logged|login/i` 识别导出器的未登录错误，英文文案 'Log in to Feishu before exporting' 命中 `login`，两种语言都能识别。
6. **商店描述**：`_locales` 只覆盖扩展名/描述/图标标题；Chrome Web Store 后台的多语言商店文案需要人工到开发者后台填写，不在代码内。
