# 交互入口与侧边栏布局重构方案

日期：2026-07-13
状态：待人工审查（未开始编码）

## 一、背景与目标

larksnap 功能已经很多（飞书导出、网页剪藏、截图、视频下载、YouTube 字幕、AI 总结、小红书卡片、公众号复制），但入口体系是各功能"长出来"的，没有统一设计过：

- 点工具栏图标只弹一个"桥接状态"popup，要再点一次按钮才能进侧边栏干活；
- 右键菜单只有网页剪藏 3 项，其余功能全都没有右键入口；
- 侧边栏按"当前页是不是飞书"二选一渲染，用户停在哪个页面决定了能看见什么功能，无法预期；
- 没有任何键盘快捷键。

本方案重新设计三个东西：**工具栏图标点击行为、右键菜单、侧边栏布局**。目标是：

1. 一次点击到达功能（消灭 popup 中转）；
2. 右键菜单覆盖高频动作，并按页面类型智能显隐；
3. 侧边栏"当前页专属功能置顶 + 通用功能常驻"，任何页面都能看到全部能力；
4. 给高频动作配快捷键。

## 二、参考扩展研究结论

研究了 4 个参考扩展（`ref/os-dist`、`ref/extracted/monica`、`ref/extracted/ai-chat-for-search`、`ref/extracted/bilibili-summary`），可借鉴的设计浓缩如下。

### 2.1 星炼助手（os-dist，Cherry Studio 浏览器版）

代码位置：`ref/os-dist/`（vite 打包产物，未混淆但已压缩）。关键文件：

- `manifest.json` — 入口、commands 快捷键、content_scripts 注册
- `assets/main.ts-DQj0KMnd.js` — 后台 service worker：图标点击分发、快捷键处理、`default_view_mode` 逻辑
- `assets/floating-button.ts-uoktljI0.js` + `assets/floating-button-BgAUX7ko.css` — 悬浮球
- `assets/selection-toolbar.ts-BJ05sO37.js` — 划词工具条（Shadow DOM、模板、自动发送）
- `assets/overlay-mPMb3ePW.js` — 页内浮窗外壳管理器（拖拽/缩放/最小化）
- `assets/viewLabels-3jbOsMo2.js` — 三形态文案与快捷键映射
- `assets/mermaid-NOHMQCX5-CBUOpDhD.js` — 主对话 App（约 1.8MB，所有 UI 字符串在此）
- 页面外壳：`sidepanel.html` / `overlay.html` / `index.html`（全窗口）/ `popup.html`（弃用）

- **同一个 App 组件复用为三种形态**：侧边栏 / 页内浮窗（iframe overlay）/ 全窗口标签页，只是 viewMode 不同；设置项 `default_view_mode` 决定点图标、点悬浮球时默认打开哪种。
- 主动 `setPanelBehavior({openPanelOnActionClick:false})` + `action.onClicked` 自己分发，popup 弃用。
- 右下角**可拖拽悬浮球**，hover 弹出三形态菜单（每项标注快捷键）；可关闭。
- **划词工具条**（Shadow DOM）：引用/翻译/解释/总结/复制五个动作，模板可配，可设"划词后自动发送"。
- 4 个键盘快捷键分别直达三形态 + 新建会话。
- 教训：申请了 `contextMenus` 权限但完全没实现右键菜单——权限要与实现对齐。

### 2.2 Monica（v9）

代码位置：`ref/extracted/monica/`（由 `ref/monica.zip` 解压，压缩混淆产物，功能靠 CSS 类名和字符串还原）。关键文件：

- `manifest.json` — action/popup、side_panel、commands（Ctrl+M、run-monica-on-new-tab）
- `background.js` — `contextMenus.create` 三组菜单、`setPanelBehavior` 可配置逻辑、`openChatTab`
- `content.js` — 主导航 tab、悬浮球 `sidebar-entry-btn`、划词工具条触发模式、reading/writing 动作、页面总结
- `content.css` / `monicaPopup.css` — CSS-Module 哈希类名（组件/功能划分的主要证据）
- 页面外壳：`monicaPopup.html`（独立打包）/ `sidePanel.html` / `chatTab.html` / `monicaOptions.html`（后三者共用 content.js）
- `static/` — 功能佐证素材（`contextual-toolbar-guide.webp`、`*SidebarExample.png`、`youtubeSummaryExample.png` 等）

- **三档形态梯度**：工具栏 popup（完整助手的紧凑版）→ Chrome 原生侧边栏 → 全屏 chatTab，承载同一套多 Tab 界面（Chat/Search/Image/Write/Read/Translate/Memo/Bots）。
- **点图标行为可配置**：`setPanelBehavior({openPanelOnActionClick: 用户设置})`，用户自己选"点图标开 popup 还是开侧边栏"。
- 右键菜单分三组、按 contexts 区分：整页翻译（page）、划词快捷指令（selection，带子菜单+超限折叠到"更多"）、图片操作（image）；随语言切换重建。
- 页面轻入口：贴边悬浮球（左右可选、按站点禁用、带未读红点）+ 划词工具条（触发模式 auto/manual 可配）。
- 页面总结按**页面类型分场景**：网页/视频/PDF/GitHub 各走专用管线，入口在 Reading tab、划词工具条、右键菜单三处会合。

### 2.3 AI Chat for Search

代码位置：`ref/extracted/ai-chat-for-search/`（由 `ref/ai chat fro search.zip` 解压，Monica 团队的 ChatGPT-for-Google 型产品）。关键文件：

- `manifest.json` — 约 190 个搜索引擎域的 content_scripts 匹配、omnibox 关键词 `gpt`、side_panel
- `content.js` — 各搜索引擎的注入选择器适配表、Readability/pdf.js 正文提取、Chat Page 全屏渲染
- `background.js` — omnibox 处理、`scripting.executeScript` 按需注入、Reading/Summarize 分发
- `sidePanel.js` — 侧边栏对话（会话列表、消息流、多模型切换、提示词库）
- `popup.js` — 启动菜单（Ask AI / 总结本页 / 开侧栏等快捷动作分发）
- `options.js` — triggerMode 三档触发模式、searchPrompt 模板、各引擎开关

- **popup 定位是"启动菜单"**：一组快捷动作（Ask AI / 总结本页 / 新对话 / 开侧边栏 / Chat Page / 提示词库 / 设置），点了就分发到对应形态，popup 本身不承载长对话。这个定位对 larksnap 很有参考价值。
- 搜索结果页右侧注入 AI 回答卡片：每个搜索引擎一份选择器适配，**触发模式三档**（总是 / 问号结尾 / 手动点按钮）——"自动但可收敛"的分寸感值得学。
- 任意页面总结：content script 不静态注入，由 background `scripting.executeScript` **按需注入**，用 Readability 提取正文——与 larksnap 现有 webcopy 动态注入思路一致。
- 完全没用右键菜单（入口靠 popup/悬浮按钮/侧栏），说明右键不是必需品，但 larksnap 的剪藏场景（对着页面/选区操作）天然适合右键。

### 2.4 BiliBili 摘要

代码位置：`ref/extracted/bilibili-summary/`（由 `ref/bilibili摘要.zip` 解压，由 ChatGPT for Google 二次改造）。关键文件：

- `manifest.json` — 权限（storage/webRequest）、匹配 bilibili.com 与 openai.com
- `content-script/index.js` — `#danmukuBox` 锚点注入、MutationObserver 触发、B 站字幕三段式 API、Prompt 模板、openai.com 页 textarea 自动化
- `background/index.js` — ChatGPT 网页会话通道（accessToken + backend-api/conversation SSE）与 OpenAI API 通道、Port 流式中转
- `popup/index.js` — 登录态检查、Cloudflare 提示
- `options/index.js` — provider 选择（网页会话 vs API key + 模型）

- 在视频页右侧栏（弹幕框上方）**页内注入摘要面板**，MutationObserver 等锚点出现即注入，挂载后自动拉字幕生成摘要，零点击。
- 字幕走 B 站公开 API（view → player/v2 → subtitle_url 三段式），不是拦截请求。
- 面板标题栏有"跳到 ChatGPT 网页继续聊"的逃生通道。
- 对 larksnap 的启发：**视频/长文摘要类功能，页内伴随面板比侧边栏更贴场景**——但这是后续可选项，不进本次范围。

## 三、现状问题清单

来自对 `src/`、`manifest.json` 的梳理（详细证据见各文件行号，此处只列结论）：

1. **popup 与功能割裂**：`Popup.tsx` 只展示桥接状态，功能要经"打开侧边栏"按钮二次进入；`background/index.ts:63` 还显式关掉了 `openPanelOnActionClick`。桥接状态只对 CC daemon 用户有意义，却占据了最贵的入口。
2. **右键菜单覆盖极不均衡**：`webcopy.ts` 只建了整页转 MD / 选区转 MD / 解除复制限制 3 项；飞书导出、截图、视频、字幕、AI 总结全无右键入口。
3. **侧边栏按页面类型二选一**：飞书页看不到剪藏/截图，非飞书页看不到飞书导出，功能可见性依赖"用户停在哪"。
4. **功能藏得深**：视频下载要"探测到视频站点且 daemon 就绪"才出现；截图埋在 WebCopyView 中部；用户无从发现。
5. **飞书 ACTIONS 11 张卡片平铺**：`word` 是永久 disabled 占位，`diagnostic`、`feedback` 与核心导出混在一起，主次不分。
6. **缓存双入口**：「缓存到本地」和「查看缓存」是两张独立卡片。
7. **小红书/公众号样式选择要两次点击**：先点卡片展开 theme-picker 再点样式。
8. **「写入飞书文档 / 采集到多维表格」无任何 UI 入口**：只能靠 daemon（larksnap-fetch skill）触发。
9. **无键盘快捷键**：manifest 没有 `commands`。
10. **header 占位元素**：「免费」徽标和 quota-chip 是无业务含义的静态占位。

## 四、重构方案

### 4.1 工具栏图标点击：直开侧边栏

- `manifest.json` 删掉 `action.default_popup`；background 改为 `setPanelBehavior({ openPanelOnActionClick: true })`。
- 点图标一步进侧边栏，与 Monica 的推荐配置一致。不做"点图标行为可配置"（YAGNI，我们只有侧边栏一种主形态）。
- popup 的桥接状态信息迁到侧边栏：header 放一个**状态圆点**（绿=已连接 daemon，灰=未连接，黄=连接中），点击展开一行详情（daemon 版本 / contextId 可复制），等价替代原 popup 全部信息。
- `popup.html`、`src/popup/` 整体删除，vite 构建入口同步移除。

### 4.2 右键菜单：一个父菜单 + 按页面类型显隐

全部挂在一个父菜单 `LarkSnap` 下（避免污染用户右键一级菜单），子项按 contexts 和 URL 分组。菜单注册收敛到一个新模块 `src/background/context-menus.ts`（现有 `webcopy.ts` 里的 `registerMenus` 迁入），仍保持"removeAll + create 幂等重建、随语言切换重建"的现有做法。

**通用网页（page context，非飞书域）：**

| id | 标题 | 行为 |
|---|---|---|
| `page-md` | 整页转 Markdown | 现有逻辑不变 |
| `selection-md` | 选区转 Markdown（仅 selection context） | 现有逻辑不变 |
| `screenshot` | 整页截图 | 发 `EXPORT_SCREENSHOT`，**右键固定 PNG**（当前配置没有截图格式字段，不引入新配置项） |
| `summarize` | AI 总结本页 | **只发导航意图**：打开侧边栏 + 定位到总结卡片 + 自动开始；意图经 storage 传递，消除侧栏加载竞态（评审意见 2） |
| `unlock` | 解除复制限制 | 现有逻辑不变 |
| `open-panel` | 打开 LarkSnap 面板 | `sidePanel.open` |

**飞书页（用 `documentUrlPatterns` 匹配飞书三域 + 已授权私有化域）：**

| id | 标题 | 行为 |
|---|---|---|
| `feishu-md` | 导出为 Markdown | 等价侧边栏 markdown 动作 |
| `feishu-pdf` | 导出为 PDF | 等价侧边栏 pdf 动作 |
| `open-panel` | 更多导出方式… | 打开侧边栏 |

说明：
- 飞书只放 md/pdf 两个最高频项，小红书/公众号等需要选样式、看预览的动作不适合右键，引导去侧边栏。
- 视频下载**不进右键**：它依赖 daemon 探测结果，右键菜单是静态注册的，做"探测到才显示"需要频繁 update 菜单，复杂度不值（KISS）。
- 私有化域授权变更时（`permissions.onAdded/onRemoved`）需要重建菜单，更新 `documentUrlPatterns`。
- 进度与结果反馈（已定案，不留到实施时再选）：**badge + 点击图标打开侧边栏查看详情**。badge 只表达状态（绿✓完成 / 红!失败），失败原因保存在**按 tab 隔离的任务记录**（storage）里，侧边栏状态栏读取展示；badge 在用户打开侧边栏或切换到该 tab 后清除；同一 tab 并发任务后者覆盖前者的 badge，任务记录各自保留。不使用 `chrome.notifications`（省权限，且 badge 方案已能承载失败详情）。

### 4.3 键盘快捷键（manifest commands）

Chrome 限制 4 个 suggested_key，正好：

| command | 建议键 | 行为 |
|---|---|---|
| `open-panel` | Ctrl/Cmd+Shift+L | 打开侧边栏 |
| `page-md` | Ctrl/Cmd+Shift+M | 当前页转 Markdown 并复制 |
| `screenshot` | Ctrl/Cmd+Shift+S | 整页截图 |
| `summarize` | 不设默认键 | 同右键：只发"开侧栏+定位+自动开始"的导航意图（用户可自行去 shortcuts 页配键） |

background 加 `commands.onCommand` 分发，逻辑与右键菜单同一套处理函数（DRY：右键、快捷键、侧边栏按钮三个入口收敛到同一个 action 分发层）。

**tabId 协议（评审意见 1，阻断项）**：`onCommand` / `contextMenus.onClicked` 触发的一瞬间就捕获 `tab.id` 和 `url`，此后全链路显式传递——截图、总结、网页转换、飞书导出的底层实现（如 `exporters/screenshot.ts`、`summarize/`）目前会自行重新查询"当前活动页"，必须改为接收调用方传入的 `tabId/url`，否则用户触发后切页会操作错标签页。这是分发层重构的第一验收标准。

### 4.4 侧边栏布局：上下文区置顶 + 通用工具常驻

**核心原则**：不再"飞书页/非飞书页二选一"，改为**单列滚动、分组卡片**——顶部是"当前页能做什么"（随页面类型变化），下面是"任何页面都能做什么"（常驻），列表项固定可预期。

```
┌─────────────────────────────┐
│ LarkSnap        ●daemon ⚙ 🗂 │  ← header：logo、daemon状态点、设置、缓存库
├─────────────────────────────┤
│ ▍当前页                       │  ← 上下文区（随页面类型变化）
│ ┌─────────────────────────┐ │
│ │ 📄 xxx文档标题（飞书 docx）│ │  ← 页面识别条：图标+标题+类型
│ └─────────────────────────┘ │
│ [导出 Markdown] [导出 PDF]    │  ← 主动作：大按钮
│ [导出 HTML]    [导出附件]     │
│ 转发布：小红书卡片 ▾ 公众号 ▾  │  ← 样式下拉直选，一次点击
│ 缓存到本地                    │
├─────────────────────────────┤
│ ▍通用工具（常驻，可折叠分组）   │
│ ▸ 网页剪藏   整页MD 选区MD 复制链接│
│ ▸ 截图       PNG / PDF        │
│ ▸ AI 总结    （未配置时显示引导）│
│ ▸ 页面开关   解除复制限制/自动复制│
├─────────────────────────────┤
│ 状态栏：进度/结果/错误          │  ← 底部固定
│ 反馈 · 诊断 · v0.x            │  ← 页脚小字链接
└─────────────────────────────┘
```

**上下文区按 pageKind 渲染：**

| pageKind | 上下文区内容 |
|---|---|
| `feishu` | 页面识别条 + 导出四主按钮 + 转发布（小红书/公众号）+ 缓存到本地 |
| `youtube` | 页面识别条 + 字幕卡（TranscriptCard）+ 视频下载卡（daemon 就绪时） |
| 视频站点（B站/抖音等） | 页面识别条 + 视频下载卡（需扩展页面模型：现有 pageKind 只有 `feishu | youtube | generic | restricted`，新增 `video` 或改为 `{kind, videoSite?, title, url}` 结构，见 `src/shared/types.ts`，评审意见 3） |
| `generic` | 页面识别条（网页标题）；剪藏动作本身在通用区，不重复 |
| `restricted` | 提示条"此页面不可操作"，通用区置灰 |

**关键改动明细：**

1. **飞书 11 卡片 → 4 主按钮 + 1 行转发布 + 1 项缓存**：
   - `word`（永久 disabled 占位）直接删除，等功能真做了再加回来；
   - `diagnostic`、`feedback` 移到页脚小字链接；
   - 「缓存到本地」留在上下文区，「查看缓存」合并到 header 的缓存库图标（🗂），CacheView 仍是二级页面（带返回）；
   - 小红书/公众号改为**下拉直选样式**（点开箭头看样式名+缩略预览，点样式即执行），消灭"先展开再选"的两段式。
2. **通用工具区常驻**：飞书页也能看到剪藏/截图/AI 总结（对飞书页执行剪藏就是对当前 DOM 走通用管线，行为不变，只是不再隐藏入口）。各分组可折叠、记住折叠态（storage）。
3. **AI 总结卡**升级为通用区常驻项：未配置端点时显示一句引导+去设置按钮（现有逻辑），配置后任何 http 页可用；YouTube 页自动带字幕上下文（现有逻辑不变）。
4. **header**：删掉「免费」徽标和 quota-chip 占位；换成 daemon 状态点（见 4.1）+ 设置 + 缓存库三个图标。
