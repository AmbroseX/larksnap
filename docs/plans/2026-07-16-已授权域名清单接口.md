# 已授权域名清单接口（new-doc 不用每次喂链接）

- 日期：2026-07-16
- 状态：待审查
- 关联：larksnap-fetch 技能 / 扩展 bridge / new-doc 新建文档

## 一、要解决的问题

现在用 `new-doc` 新建飞书文档，每次都得手动传一个「该域名下的链接」。原因是：

- `new-doc` 的第一个参数只是用来告诉扩展「在哪个飞书租户/域名下建」（见 `skills/larksnap-fetch/scripts/edit.mjs:113`）；
- 公有云还能猜，但私有化部署是**用户自建域名**，脚本没地方记住是哪个，所以只能每次现喂一个链接。

尤其是同时用多个飞书（公有云 + 私有域名 A + 私有域名 B）时，来回喂链接很烦。

## 二、核心思路：向扩展要「已授权域名清单」，而不是自己另存一份

扩展本来就持有登录态和已授权域名，**数据源现成，零维护**：

- **私有域名**：用户在侧边栏点「授权该域名」后，Chrome 把 `*://*.域名/*` 加进 `optional_host_permissions`，扩展的 `listTrusted()`（`src/background/permissions.ts:48`）已经能返回这批 pattern（存在扩展 storage 的 trustedDomains）。
- **公有云**：`feishu.cn` / `feishu.net` / `larksuite.com` 是 `manifest.json` 里写死的 `host_permissions`，直接拼上即可。

好处：

1. 用户在侧边栏授权一个新私有域名，它**自动**出现在清单里，不用改任何配置文件；
2. 撤销授权后也自动从清单消失，永不和实际授权状态失同步；
3. 不引入独立的「域名配置文件」，避免两处数据打架。

**但有一个绕不开的坑（决定了本方案的一半得靠额外数据）**：`listTrusted()` 存的是**基础域通配** `*://*.{基础域}/*`（授权时 `permissionPattern()` 主动剥掉了租户子域，见 `src/shared/feishu-host.ts:11-18`）。而 `new-doc` 要开一个**能真正加载、能建文档的租户页面**，需要的恰恰是被剥掉的租户子域——光有基础域 `corp.example.com`，合成 `https://corp.example.com/drive/me/` 会打到 apex（公有云是营销站、私有化未必是网盘 host），建不出文档。

所以「列出域名」这半边基础域够用；「合成一个能建文档的 landing URL」这半边必须补一份**真实 origin**。采用 **A 为主 + B 兜底**：

- **A（主）**：用户在侧边栏授权私有域名的**那一刻**，把他当时所在页面的真实 origin（如 `https://tenant.corp.example.com`）顺手记进扩展 storage——这是**丰富现有授权记录**，不是第二份独立配置，撤销授权时一并清掉。
- **B（兜底）**：storage 里没有（老授权、或公有云这种没有「授权手势」的内置域名）时，扩展当场 `chrome.tabs.query` 找一个 host 匹配该基础域的**已打开标签**，取它的 origin。
- 两者都没有 → CLI 让用户**这一次**喂个 URL，优雅降级，不静默失败。

## 三、要改的地方（自底向上）

### 3.0 扩展授权存储：授权时顺手记下真实 origin（方案 A）

现状：侧边栏授权走 `REQUEST_PERMISSION`（`src/background/index.ts:164-169`），只把 `permissionPattern(host)` 这条**通配 pattern** 存进 `trustedDomains`（`src/shared/storage.ts`），租户子域丢了。

改动：授权成功那一刻，侧边栏本来就有当前页 `host`（`SidePanel.tsx:193` 上下文），顺手把真实 origin 一并带上并存下来。

- 新增一个小 map：`trustedOrigins: { [基础域]: 'https://<真实租户 host>' }`（键用 `baseDomain(host)`，和 `trustedDomains` 对齐）。
- 写入点：`REQUEST_PERMISSION` 里 `recordTrusted(pattern)` 旁边加一句 `recordTrustedOrigin(baseDomain, origin)`。
- 清理点：撤销授权 `revokePermission` 里一并删掉对应 key，绝不残留。
- 这是**在现有授权记录上加一个可选字段**，不是独立配置；老授权没有这条 → 走 B 兜底。

### 3.1 扩展 `src/background/bridge.ts`：新增 `list-domains` 消息分支

落点在 onMessage 分发处（`src/background/bridge.ts:400` 那段 `msg.type === 'job'` 附近）。现有 job 都要求带 `url`，列域名不需要 url，是个独立的新消息类型。

伪代码：

```ts
// onMessage 里，job 分支之前/之后加：
if (msg.type === 'list-domains' && msg.id) {
  const trusted = await listTrusted();          // 私有域名 pattern：*://*.xxx.com/*
  const builtin = ['feishu.cn', 'feishu.net', 'larksuite.com'];
  const domains = await buildDomainList(builtin, trusted); // 内部为每个基础域解析 sampleUrl
  reply(msg.id, { type: 'domains-result', domains });
  return;
}
```

`buildDomainList` 把两类来源归一化成统一结构（见 §4 数据结构），并为每个基础域解析出 `sampleUrl`（能建文档的真实 origin），**按优先级**：

1. **A**：`trustedOrigins[基础域]` 存过的真实 origin；
2. **B**：`chrome.tabs.query({})` 里 host 命中该基础域（`host === base || host.endsWith('.'+base)`）的已打开标签的 origin；
3. 都没有 → `sampleUrl: null`（CLI 侧再决定是报错还是让用户喂 URL）。

需要 `import { listTrusted, listTrustedOrigins } from './permissions'`。

### 3.2 daemon `scripts/bridge/daemon.mjs`：新增只读端点 `GET /hosts`

daemon **不是**通用透传：`/command` 入口硬性 `if (!job.url) 400`（`daemon.mjs:174`），且只往扩展发 `type:'job'`（`daemon.mjs:206`），bridge 侧也卡 `msg.url`（`bridge.ts:400`）。一条不带 URL 的 `list-domains` 会在这三层各被挡一次。

所以 daemon 必须改：**新增一个独立的只读端点 `GET /hosts`**，绕开 `/command` 的 url 校验，内部向扩展发 `list-domains`、收 `domains-result` 后回 HTTP。具体 HTTP 契约见 §4.2。

### 3.3 CLI：新增 `hosts` 命令（列清单）

在 `skills/larksnap-fetch/scripts/edit.mjs`（或独立小脚本）加一个只读命令：

```bash
node .../edit.mjs hosts                    # 打印已授权域名清单（单 profile）
node .../edit.mjs hosts --profile <code>   # 多浏览器 profile 时指定其一
```

输出 JSON：`{ ok: true, domains: [ { host, kind } ] }`（字段定义见 §4）。多 profile 且不带 `--profile` 时报 `profile_ambiguous`。

### 3.4 CLI `new-doc`：真正省掉喂链接（本计划的核心闭环）

现状（`skills/larksnap-fetch/scripts/edit.mjs:105`）：`if (!url || !op)` 硬性要求第一个定位参数是 URL。要让「不喂链接」成立，得把 `new-doc` 的参数放宽：

**新的用法**（其他操作不变，仍要求 URL）：

```bash
# 1) 一个域名：直接建
node .../edit.mjs new-doc 内容.md --name "标题"

# 2) 多个域名：显式选一个基础域（或它的唯一后缀简写）
node .../edit.mjs new-doc 内容.md --name "标题" --host a.example.com
node .../edit.mjs new-doc 内容.md --name "标题" --host feishu.cn

# 3) 还是想传 URL：完全兼容旧用法
node .../edit.mjs new-doc https://xxx.feishu.cn/drive/me/ 内容.md --name "标题"
```

CLI 内部逻辑（伪代码）：

```
if op == 'new-doc' && positionals[0] 不是 http(s) URL:
  domains = 调 daemon /hosts 拿清单（每项含 host / kind / sampleUrl）
  if flags['--host']:
    picked = 在 domains 里按 host 精确匹配，找不到再按后缀唯一匹配
    picked 不存在 → 报 host_not_authorized（附上 domains 让用户选/去授权）
  elif len(domains) == 1:
    picked = domains[0]
  else:
    报 host_ambiguous（附上 domains 让用户加 --host）

  # 关键：landing URL 用 picked.sampleUrl（真实租户 origin），不是拿基础域硬拼
  if picked.sampleUrl:
    url = `${picked.sampleUrl}/drive/me/`   # sampleUrl 已是真实租户 origin
  else:
    报 need_landing_url（提示：该域名下没有已记录的 origin，也没开着相关标签；
                        请这一次喂个该域名下的文档/网盘链接，或先在浏览器打开一个该域名页面）
  positionals 前插 url，让下游 new-doc 分支保持原样走
```

**为什么必须用 sampleUrl，不能拿基础域硬拼**——这是本方案的地基：

`createEmptyDoc`（`src/background/editor.ts:283`）不是只取 host，而是真的 `chrome.tabs.create({ url: job.url })` 打开这个 URL、`waitForTabComplete` 等它加载完、注入 `content.js`、再在该页上下文里跑 `createDocx`（`feishu-api.ts:111` 用**相对路径**调 `/space/api/explorer/v2/create/object/`，依赖该页登录态 + CSRF）。所以 landing URL 得是一个**能加载出编辑器上下文、能注入脚本、能建文档**的真实租户页面。

- 基础域 apex（`https://feishu.cn/...` 或私有化 `https://corp.example.com/...`）不满足这个条件——这正是 §二那个坑，也是为什么要 A/B 补一份真实 origin。
- `sampleUrl` 已是真实租户 origin（A 存的授权页 origin，或 B 抓的已开标签 origin），拼上网盘首页路径即可。
- `/drive/me/` 这个路径本身来自 SKILL.md 现有 new-doc 提示（「网盘首页 `/drive/me/`」），**仍需实测确认**在真实 origin 上能成功建文档（见 §5 手测最高风险项）。

改动收敛在 CLI + 扩展 storage/list-domains，不碰 `createEmptyDoc` 的建文档逻辑。

其他操作（append/insert-after/list-blocks/…）**继续要求真实文档 URL**，因为它们需要具体文档 token，`--host` 对它们没意义——CLI 层直接拒绝这些操作带 `--host`，避免用户误用。

## 四、数据结构

### 4.1 扩展→daemon→CLI 的域名清单

```json
{
  "ok": true,
  "domains": [
    { "host": "feishu.cn",     "kind": "builtin", "sampleUrl": null },
    { "host": "feishu.net",    "kind": "builtin", "sampleUrl": null },
    { "host": "larksuite.com", "kind": "builtin", "sampleUrl": null },
    { "host": "a.example.com", "kind": "trusted", "sampleUrl": "https://tenant.a.example.com" },
    { "host": "b.example.com", "kind": "trusted", "sampleUrl": null }
  ]
}
```

字段说明（严格 YAGNI，不引入 alias/default 这类需要额外存储的字段）：

- `host`：**基础域**（如 `a.example.com`），不是 pattern。扩展侧从 `listTrusted()` 的 `*://*.a.example.com/*` 里剥出来；`builtin` 直接是 manifest 里的字面量。
- `kind`：`"builtin"` = manifest 内置三域名；`"trusted"` = 用户手势授权的私有域名（来自 `listTrusted()`）。区分它是为了 UI/日志能标注来源，不参与选择逻辑。
- `sampleUrl`：该基础域下**一个能建文档的真实租户 origin**（`https://tenant.a.example.com`），或 `null`。解析优先级见 §3.1：A（授权时存的 origin）→ B（当前已打开标签）→ null。CLI 用它拼 landing URL（§3.4），**这是「省链接」真正落地的字段**。
  - 私有域名（trusted）常有值（用户授权过 → A 命中）。
  - 内置域名（builtin）多为 `null`：公有云没有「授权手势」可捕获 origin，且 apex `feishu.cn` 不是用户租户；只有当用户正开着某个 `*.feishu.cn` 标签时 B 才补得上。这是公有云 new-doc「省链接」不如私有域名稳的**已知局限**，写进 SKILL.md。
- **这份清单是「某一个 profile」的授权域名，不是全局并集**：daemon 按 `contextId`（浏览器 profile）维护连接，各 profile 授权的私有域名可能不同。多 profile 时 `/hosts` 必须带 `--profile` 指定其一（详见 §4.2），返回的就是那个 profile 扩展的清单。
- **不做 alias**：`--host` 直接收 host 或它的唯一后缀简写（`a` 匹配 `a.example.com`，多义就报错）。别名要新增存储、要 UI 维护，YAGNI。
- **不做 default**：一个域名时天然是默认；多个域名时强制 `--host`。省掉一层「谁是默认、什么时候变」的心智负担。用得多了再回来加。

### 4.2 daemon HTTP 契约

新增只读端点，独立于 `/command`，绕开它的 url 校验（这是评审二的采纳方案）：

- 请求：`GET /hosts?profile=<code>`（`profile` 可选）。带与 `/command` 相同的**两个头**：`x-larksnap`（`AUTH_HEADER`）+ `x-larksnap-sig`（`SIG_HEADER`）。注意 `verifySigHeader` 把 `method + pathname + body` 一起纳入签名（`daemon.mjs:132`），CLI 现在只有 POST 侧签名（postCommand），**得新增一条 GET 签名调用**，method 用 `GET`、body 为空，两边算法保持一致。
- **profile 路由**：daemon 用 `resolveConn(profile)` 找扩展连接（`daemon.mjs:55`）。不带 profile 且只连了一个扩展时用那个；多 profile 且不带 profile 时返回 `profile_ambiguous`，提示加 `--profile`。
- 响应：`200 application/json`，body 就是 §4.1 的 JSON。
- 错误（沿用现有 subtype，均 `200` + `{ok:false, error:{subtype}}`，签名失败除外）：签名/鉴权失败走现有 `403`（`daemon.mjs:120/147`）、`extension_not_connected`、`profile_not_found`、`profile_ambiguous`、`extension_outdated`。
- 为什么用 GET 独立端点而不是复用 `/command`：`/command` 的整条链路（400 拒 url-less、只发 `type:'job'`、bridge 也卡 url）都是为「带 URL 的任务」设计的，为一条只读消息把三层校验都改例外，不如新开一个只读小端点干净——评审二的判断成立，本计划直接采纳。

### 4.3 协议版本与向后兼容

- **协议版本要升到 v4**：`PROTOCOL_VERSION` 现值**已经是 3**（`bridge.ts:43`、`protocol.mjs:21`，v3 = 扩展主动发 video-job），不能再叫 v3。本计划把两个常量**一起 bump 到 4**（`list-domains`/`domains-result`）。`daemon.mjs:185` 的 `_proto < 2` 是 edit 专用老门槛，跟本特性无关，别复用。
- daemon `/hosts` 在派发前检查 `_proto >= 4`，否则回 `extension_outdated`，让用户更新扩展——参考现有 edit 任务的降级模式（daemon.mjs:185）。
- bump 版本后，装着 v3 扩展的老用户会触发 `bridge.ts:380` 的「桥接协议版本不一致」警告，直到重建扩展；这是预期行为，SKILL.md 里提示「重新构建/加载扩展」即可。
- CLI 拿到 `extension_outdated` 时，new-doc 走「无链接」分支的会给出明确提示：**要么更新扩展，要么按旧用法喂一个链接**。不静默失败。
- **个人版 `my.feishu.cn`** 是 `*.feishu.cn` 的子域，权限上被 builtin 覆盖，功能上接口有差异（见 memory `feishu-personal-my-feishu-cn`）。本期**不在数据结构里标注**，只在 SKILL.md 提示：`--host my.feishu.cn` 目前不保证 new-doc 一次成功，用户遇到再按提示走。留标记位属于过度设计，YAGNI。

## 五、落地任务清单（按依赖顺序）

1. **扩展授权存储（方案 A）**：`storage.ts` 加 `trustedOrigins` map 及 `recordTrustedOrigin`/`listTrustedOrigins`/删除三个函数；`index.ts` 的 `REQUEST_PERMISSION` 授权时存真实 origin、`REVOKE_PERMISSION` 撤销时一并删。
2. **扩展 `src/background/bridge.ts`**：加 `list-domains` onMessage 分支，`buildDomainList` 拼 builtin + trusted 并按「A→B→null」解析每项 `sampleUrl`（B 用 `chrome.tabs.query`），回 `domains-result`。`PROTOCOL_VERSION` 升到 4（`bridge.ts:43`、`protocol.mjs:21` 一起改）。
3. **daemon `scripts/bridge/daemon.mjs`**：新增 `GET /hosts` 只读端点，鉴权+签名校验（`x-larksnap` + `x-larksnap-sig`，GET 空 body）→ `resolveConn(profile)` 找连接 → 检查 `_proto >= 4` → 发 `list-domains` → 收 `domains-result` → 回 HTTP JSON。含 `extension_outdated`/`extension_not_connected`/`profile_ambiguous`/`profile_not_found` 分支。
4. **CLI `skills/larksnap-fetch/scripts/edit.mjs`**：新增 `hosts` 命令（只读打印清单，支持 `--profile`，含 GET 签名调用）；`new-doc` 放宽第一参数校验，实现 §3.4 的选域名/用 `sampleUrl` 合成 URL 逻辑（`--host` + `--profile`），无 `sampleUrl` 时报 `need_landing_url`；其他操作显式拒绝 `--host`。
5. **SKILL.md**：更新 new-doc 段落，把「省链接」用法写清楚；说明私有域名比公有云更稳（sampleUrl 命中率）、无 origin 时要喂一次 URL；补 `my.feishu.cn` 已知限制；提示 bump 版本后需重建扩展。
6. **手测**：
   - **合成 URL 能成功建文档**（最高风险项，先测）：私有域名授权后 `new-doc <内容.md> --host <域名>` 用 `sampleUrl`（方案 A）合成路径，真能打开页面、注入脚本、建出文档；
   - **方案 B 兜底**：清掉某域名的 `trustedOrigins` 记录、只开着该域名一个标签，确认 `hosts` 仍能给出 `sampleUrl` 并建文档成功；
   - **无 origin 降级**：既无记录又无标签时，报 `need_landing_url` 而不是静默失败；
   - 单域名一次、多域名带 `--host` 一次、多域名不带 `--host` 报 `host_ambiguous` 一次；
   - 多 profile 带 `--profile` 一次、不带报 `profile_ambiguous` 一次；
   - 撤销授权后该域名从 `hosts` 清单消失、`trustedOrigins` 记录也清掉；
   - 旧扩展（v3）降级到 `extension_outdated` 一次；旧用法（喂真实 URL）兼容一次。
   - **按 CLAUDE.md 规矩，用户手测通过 + 明确要求后再 commit。**

## 六、非目标（明确不做）

- 不做别名、不做默认域名、不做独立的 config.json——一旦引入就要新的写入时机、UI、迁移，全违反 YAGNI。
- **会给授权存储加一个 `trustedOrigins` 小 map（方案 A）**，但只在现有授权/撤销时机顺手读写，不新增 UI、不做迁移（老授权没有就走 B 兜底）——这是本计划从「零改存储」修正后的结论，见 §二/§3.0。
- 不在扩展里凭空合成 landing URL：扩展只负责**提供** `sampleUrl`（真实 origin），拼路径、选域名的脏活交给 CLI，改动收敛。
- 不为个人版做特殊标记位，只在文档里提示。

## 七、后续可选增强：让公有云也做到「零链接」（本期不做）

**动机**：本期方案下，公有云（`feishu.cn` 等 builtin）的 `sampleUrl` 命中率不如私有域名——公有云没有「授权手势」这个捕获点（A 抓不到），apex 又不是用户租户，只有当用户正开着某个 `*.feishu.cn` 标签时 B 才补得上。所以公有云下如果没开着飞书标签，还是得喂一次 URL。

**增强思路**：把方案 A 的 origin 捕获点，从「只在授权手势时」扩展到「**任意一次成功操作过某飞书页面时**」——扩展在跑完抓取/编辑任务、确认某个 tab 是真实登录的飞书租户页后，顺手把它的 origin 记进 `trustedOrigins[基础域]`（键仍用 `baseDomain(host)`）。这样用户只要**用过一次**某个公有云租户，之后 new-doc 就能零链接。

**为什么本期不做**：

1. 捕获点变多（要在任务成功路径上插一手），改动面比「只在授权时记一次」大，违背 YAGNI——先把私有域名这条主线跑通、真机验证「省链接」体验，再决定要不要铺开。
2. 公有云本来就有 B 兜底（开着标签就行），痛点没私有化那么尖。
3. 有隐私/正确性细节要想清楚：多租户用户会在同一基础域下覆盖 origin（`trustedOrigins` 只存一个），得定义「记最后一次用的」还是「按租户分别存」——这属于 A 单值 map 撑不住时才需要的复杂度，不急。

**落地时的小口**：真要做，捕获逻辑收在扩展任务成功回调里加一句 `recordTrustedOrigin(baseDomain(host), origin)` 即可，数据结构（`sampleUrl`）和 CLI 侧完全不用动——本期的 §3.1/§3.4/§4.1 已经为它留好了位置。
