# 参考项目借鉴分析：ref/cli 与 ref/lark-cli

> 日期：2026-07-03
> 状态：分析结论，待人工审查后拆解为具体开发计划
> 调研对象：`ref/cli`（飞书官方 lark-cli，Go）、`ref/lark-cli`（第三方 lark-cli，Go）

## 一句话结论

两个参考项目都是"注册开放平台应用 + OAuth + 官方 API"路线的 Go CLI，和 larksnap 的"复用浏览器登录态 + 网页内部接口"路线完全不同，**API 调用层的代码抄不了**。真正值得借鉴的是三块：

1. daemon 桥接的安全设计（HMAC 签名、协议版本握手）
2. 面向 AI 的错误/输出契约（结构化错误、部分失败信封、有界运行）
3. SKILL.md 的组织方式（示例输出、多步工作流、报错自愈指引）

## 技术路线对比

| | larksnap | ref/cli（官方） | ref/lark-cli（第三方） |
|---|---|---|---|
| 形态 | 浏览器扩展 + 本地 daemon | Go CLI + 可选 sidecar | Go CLI |
| 登录态 | 浏览器 Cookie | OAuth（keychain 存凭据） | OAuth（tokens.json） |
| 接口 | 飞书网页内部接口 | 开放平台 API（2500+） | 开放平台 API |
| 文档转 MD | 自己解码 apool + block 树 | — | 调官方 markdown 导出接口 |
| 私有化域名 | 支持（核心卖点） | 不支持 | 不支持 |
| 给 AI 用 | larksnap-fetch skill | 27 个内嵌 skill，Agent-Native 设计 | 8 个 skill |

关键差异：两个参考项目要注册应用、申请权限，正是 larksnap 刻意绕开的门槛。所以它们"怎么调 API"没有参考价值，"怎么把工具做得让 AI 好用、让本地进程间通信安全"才有。

## 从 ref/cli（官方 CLI）借鉴什么

它的 sidecar 模式（沙箱里的 CLI ⇄ 持有真凭据的本地可信进程）和 larksnap-fetch 的桥接（CLI ⇄ daemon ⇄ 持有登录态的扩展）是同一个问题的两种解法，它在安全和生命周期上做得更完备。

### 1. daemon 通信加 HMAC 签名（最值得抄）

**现状**：daemon 靠"校验 Origin + 要求 `X-Larksnap` 自定义头"挡 CSRF。这只能防浏览器页面发起的请求，防不了本机其他进程冒充 CLI 或冒充扩展。

**它的做法**（`ref/cli/sidecar/hmac.go`、`sidecar/protocol.go`）：
- 客户端和服务端共享一把 key，每个请求做 HMAC-SHA256 签名
- 签名覆盖：协议版本、method、host、path、body 的 SHA-256 摘要、时间戳、身份、auth 头名 —— 身份和 auth 头进签名，攻击者在重放窗口内也没法翻转身份或把凭据重定向到会被日志记录的头里
- 时间戳漂移上限 60 秒，防重放
- 地址强制 loopback（127.0.0.1/::1）或已知同机别名，拒绝 https、拒绝 URL 里带用户名密码

**多租户 demo 的进阶做法**：每个客户端一把 key，服务端靠"哪把 key 能验签"识别请求来源，无回落（回落 = 认证漏洞）。这正好对上 larksnap 已有的多浏览器 profile（contextId）路由，可升级成**每个 profile 一把 key**。

### 2. 结构化错误契约（`ref/cli/errs/ERROR_CONTRACT.md`）

**现状**：larksnap-fetch 已有退出码约定（0 成功 / 1 失败 / 2 用法错 / 3 需登录 / 4 需授权域名 / 5 桥接未就绪），错误信息是散文。

**它的做法**：错误退出时往 stderr 写**一行 JSON**：

```json
{"ok": false, "error": {"type": "authentication", "subtype": "need_login",
 "message": "哪里错了（给人读）", "hint": "下一步执行什么（命令式）", "retryable": false}}
```

- `type`/`subtype` 是闭合枚举，AI 靠机器字段分支，不解析散文
- `message` = 哪里错了，`hint` = 下一步做什么，两者分离
- 退出码由错误类别派生，不手写
- 设计律条："每条错误消息都会被 AI 解析来决定下一步动作"

**对 larksnap 的意义**：skill 本来就是给 Claude Code 用的。`need-login`/`need-auth` 带上结构化 `hint`（如"打开 xx 页面登录后重试"）和 `retryable`，agent 的自愈能力会明显变好。

### 3. 批量导出的两个现成模式（对应路线图"知识库批量导出"）

- **NDJSON 流式 + 有界运行**：`event consume` 支持 `--max-events` / `--timeout`（超时算正常退出、退出码 0）、非 TTY 时把 stdin 被关掉当停止信号、启动就绪时往 stderr 打 ready 标记。全是为"被 AI 当子进程调用"设计的。整库导出（几十上百篇、跑几分钟）可以原样搬。
- **部分失败信封**：批量导 50 篇成功 47 篇时，往 stdout 写 `ok:false` 的汇总 JSON（总数 + 每篇成败明细），而不是整体报错或整体报成功。AI 按 `data.items[]` 逐项决策重试。

### 4. 版本漂移检测

**它的做法**：JSON 输出里塞 `_notice.skills`，提醒"本地 skill 文档和 binary 版本对不上，建议更新"。

**larksnap 更容易漂移**：larksnap-fetch 自带 daemon 脚本、可装到任何项目，而扩展独立更新——协议一变，装在别处的旧 skill 就悄悄坏掉。建议：**WS 握手带协议版本号**，不匹配时返回明确的"请更新 skill"提示。

### 5. daemon 发现防竞态

**它的做法**：每个 app 一个 PID 文件 + 进程存活期的文件锁（advisory lock）。

**现状与风险**：larksnap 靠 `/ping` 探活 + 按需拉起 + 空闲 30 分钟自退。两个 CLI 同时冷启动时有竞态拉起两个 daemon 的可能。PID 文件 + 锁是零依赖的补法。

## 从 ref/lark-cli（第三方 CLI）借鉴什么

它的实现层（OAuth token 刷新、scope 管理、IMAP 缓存）跟 larksnap 无关，但"给 AI 用的产品思路"有两点做得好：

### 1. SKILL.md 的写法

它的每个 skill 是一份"给 AI 看的命令手册"：

- 每条命令带**示例 JSON 输出**（AI 看一眼就知道返回长什么样，不用试错）
- 写清**多步工作流**（"先 A 拿到 X，再用 X 调 B"，如：先 append 建表 → blocks 拿结构 → 逐格写入）
- frontmatter 的 description 用 "Use when..." 触发词，让 AI 准确路由

**对 larksnap-fetch 的补充方向**：各退出码对应的实际输出样例；"未登录 → 提示用户登录 → 重试"的完整工作流示例。

### 2. 报错即自愈指引

缺权限时直接在错误里给出可复制执行的修复命令（如 `To add these permissions, run: lark auth login --add --scopes documents`）。和上面 ref/cli 的错误契约是同一个思想，这个版本更轻量，适合先做。

### 不能抄的（记录下来防止走弯路）

它文档转 Markdown 是直接调开放平台原生导出接口（`/docs/v1/content?content_type=markdown`），一行搞定、不用解析 block。**但那是开放平台 API，要注册应用、申请权限，正是 larksnap 刻意绕开的门槛。** larksnap 的 apool 解码 + block 树转换没有捷径可替代，这条路已经走对了。

## 优先级建议

结合现状（SmartCopy P0 待真机验证、批量导出未动工）：

| 优先级 | 事项 | 理由 |
|---|---|---|
| P1 | 错误契约（stderr 一行 JSON + hint + retryable）+ SKILL.md 补示例输出和工作流 | 改动小，对 AI 调用成功率提升立竿见影 |
| P2 | daemon HMAC 签名（可按 profile 分 key）+ WS 握手协议版本号 | skill 要分发出去，安全和兼容协议越早定越便宜 |
| P3 | PID 文件 + 文件锁防 daemon 重复拉起 | 竞态窗口小，顺手修 |
| 做批量导出时再引入 | NDJSON 流式、`--timeout`/stdin-EOF 有界运行、部分失败信封 | 遵循 YAGNI，别提前做 |
