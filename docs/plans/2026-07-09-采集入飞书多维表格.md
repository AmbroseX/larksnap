# 外部内容采集入飞书多维表格 —— 实施方案

日期：2026-07-09
状态：待人工审查
来源：`docs/research/2026-07-09-参考扩展解包分析.md`（社媒助手 v3.3.2 逆向）
优先级：与 larksnap 主业（飞书）强契合，中等偏重，建议在视频下载之后或并行推进

## 一、要做成什么（第一性原理）

在一个网页（小红书笔记、抖音视频、B 站、公众号文章，或任意网页）上，让用户一键把这条内容
**归档进飞书多维表格**：标题、作者、链接、正文摘要、封面图/视频等字段各就各位，图片和视频作为
附件存进表里。用户攒下来的表就是一个可检索、可协作的"素材库/收藏夹"。

这是 larksnap 现有能力的自然延伸——larksnap 本来就干"把网页转 Markdown"（webcopy/larksnap-fetch）
和"操作飞书"（导出/编辑），把两者接起来就是"网页 → 飞书结构化存档"。

## 二、和参考扩展的关键差异：larksnap 走登录态，免去用户配密钥

社媒助手把数据写进飞书走的是**开放平台自建应用**：用户要自己在飞书后台建应用、填 app_id +
app_secret、开权限，扩展拿密钥换 tenant_access_token 再调 OpenAPI。这套对用户是重体验
（建应用、配密钥、机器人建表还要转 owner）——因为社媒助手**没有飞书登录态资产**，只能走 OpenAPI。

**larksnap 不一样：它有飞书登录态。** larksnap 现有的 `feishuGet`/`feishuPost`
（`src/background/feishu-proxy.ts`）已经能带着用户的登录 cookie + CSRF 打飞书**内部接口**
（`/space/api/...`），飞书网页版多维表格自己用的就是这套内部接口。所以 larksnap 应走**路线 B**：

| | 社媒助手（路线 A，OpenAPI 自建应用） | larksnap（路线 B，登录态内部接口） |
| --- | --- | --- |
| 鉴权 | 用户建应用 + app_id/secret → tenant_access_token | 复用现有登录态 cookie + CSRF（`feishuPost`） |
| 建表 | 机器人建表，owner 是机器人还要转给用户 | 用户身份直接建表，天然属于用户 |
| 用户负担 | 建应用、配密钥、开权限（劝退） | 已登录飞书即可，零配置 |
| 前提 | 无飞书登录态也能用 | **依赖飞书登录态**（larksnap 本就有） |

**代价**：内部接口是私有的、飞书可能改，需要先探测确认（见第四节）。但这正是 larksnap 一贯的
技术路线（client_vars、导出、编辑都走内部接口），风险与现状一致，不是新增风险。

## 三、可直接复用 / 要新增的部分

对照社媒助手的完整链路，larksnap 已有一大半：

| 环节 | larksnap 现状 | 结论 |
| --- | --- | --- |
| 采集网页内容 | webcopy `html2md` + larksnap-fetch 抓取 | 复用（普通网页）；社媒平台需加适配 |
| 登录态调飞书内部接口 | `feishuPost`/`feishuGet` 带 cookie+CSRF | 复用做地基 |
| 媒体下载 | `downloadMedia`（SW 带 cookie fetch，防盗链） | 复用 |
| 消息桥 / daemon | bridge + daemon | 复用（批量采集可让 daemon 编排） |
| 多维表格写入 | 无 | **新增（内部 bitable 接口，先探测）** |
| 媒体上传飞书素材 | 无 | **新增（内部素材上传接口，先探测）** |
| 字段映射（声明式 schema） | 无 | **新增（照搬社媒助手的 feishu:{type} 思路）** |
| 平台采集适配（小红书/抖音等） | 无 | **新增（MAIN 世界劫持 API，增量补站）** |

真正要写的是四块：**bitable 写入、素材上传、字段映射、平台适配**。前两块要先探测内部接口。

## 四、先决任务：探测飞书内部多维表格接口

写代码前必须先用 `feishu-doc-probe` 技能（Playwright 在已登录飞书页抓包）确认这几组内部接口的
真实形态（路径、参数、鉴权、返回），因为它们是私有的、无公开文档：

1. **建多维表格 App / 数据表 / 字段**：飞书网页版新建 base、加字段时发的 `/space/api/...` 请求。
2. **记录增删改查**：`batch_create` 记录时的内部接口（对应 OpenAPI 的
   `bitable/v1/apps/{app}/tables/{table}/records/batch_create`，内部版路径待测）。
3. **素材上传**：把图片/视频传进 base 附件字段时走的内部上传接口（对应 OpenAPI 的
   `drive/v1/medias/upload_all` / 分片三步，内部版待测），以及拿到的 token 怎么回填附件字段。
4. **字段类型编号**：确认内部接口的字段 type 编号是否和 OpenAPI 一致（1 文本 / 2 数字 /
   3 单选 / 4 多选 / 15 URL / 17 附件），formatter/options 结构是否一致。

**兜底**：万一某些内部接口拿不到或太不稳定（尤其素材上传），退回**路线 A（OpenAPI + 自建应用）**
只对这一步降级——即"记录写内部接口、素材传 OpenAPI"混用，或整体给用户一个"高级：配飞书应用"
的可选开关。第一版以探测结果决定，探测是这个方案的第一个里程碑，探测不通就调整范围。

## 五、数据流

```
网页点扩展"存入飞书表格"
  → 扩展：采集这条内容（普通网页走 webcopy html2md；社媒平台走 MAIN 世界劫持 API 拿结构化 JSON）
  → 扩展：归一成一条"采集记录"{ 字段值 + 媒体URL列表 }
  → SW：downloadMedia 把封面/视频抓成二进制（带 cookie/Referer 绕防盗链，复用现有）
  → SW：feishuPost 建/复用目标多维表格与字段（首次建表，之后复用用户选的表）
  → SW：内部素材上传接口传媒体拿 file_token
  → SW：feishuPost records/batch_create 写一条记录（文本/链接字段 + 附件 file_token）
  → 完成，sidepanel 显示"已存入《表名》"，给出表链接
```

批量场景（采集某博主主页全部笔记）：扩展在 MAIN 世界用平台列表 API + 游标翻页拉全清单，
交给 daemon 编排逐条走上面的流程（daemon 能长期驻留、扛几十上百条不受 SW 生命周期限制）。

## 六、中间表示：采集记录 + 声明式字段映射

照搬社媒助手最值得借用的设计——**采集模型自带飞书字段类型声明**，同步时一把生成表结构：

```ts
// src/background/collect/schema.ts
export interface FieldDef {
  key: string;            // 采集数据里的字段名
  label: string;          // 飞书表里的列名
  feishu: {
    type: 1 | 2 | 3 | 4 | 15 | 17;   // 文本/数字/单选/多选/URL/附件
    property?: unknown;              // 数字 formatter、单选 options、附件 file_extension 等
  };
}

export interface CollectRecord {
  fields: Record<string, unknown>;   // key → 值
  media: { key: string; url: string; kind: 'image' | 'video' | 'audio' }[];
}
```

每个平台适配器导出一份 `FieldDef[]`（表结构）+ 一个 `parse(): CollectRecord`（从页面数据填值）。
第一版做一套通用字段（标题/作者/链接/正文摘要/封面）+ 小红书、抖音两个平台适配，其余站
先走"普通网页"通用采集（webcopy 转 Markdown 填正文 + 页面 og:image 当封面）。

## 七、改动清单（按实现顺序）

**里程碑 0：探测（最先做，决定后续范围）**
1. 用 `feishu-doc-probe` 抓包确认第四节四组内部接口的真实形态，产出一份接口备忘录写进
   `docs/research/`。探测不通的接口标记走 OpenAPI 兜底。

**里程碑 1：单条网页存入指定表（MVP）**
2. `src/background/feishu-bitable.ts`：封装内部接口——建 App/表/字段、`batch_create` 记录、
   素材上传（都经现有 `feishuPost` 或 SW fetch）。
3. `src/background/collect/schema.ts` + 通用字段定义。
4. `src/background/collect/media-upload.ts`：`downloadMedia` 拿二进制 → 内部素材上传 →
   file_token（小文件一把传，大文件分片，照社媒助手阈值 20MB + 指数退避重试）。
5. `src/background/exporters/`（或新目录 `collect/`）：`collectToBitable(record, tableRef)` 编排。
6. UI：sidepanel/popup 在非飞书网页上加"存入飞书表格"入口；首次让用户选/建目标表并记住
   （存 `chrome.storage`）。
7. 消息：`src/shared/constants.ts` 加 `MSG.COLLECT_TO_BITABLE`；`index.ts` 加 case。

**里程碑 2：平台适配 + 批量**
8. `src/content/collect/xiaohongshu.ts`、`douyin.ts`：MAIN 世界劫持平台 API（和视频下载的取址
   同源，抖音白嫖签名），产出结构化 `CollectRecord` + 平台专属 `FieldDef[]`。
9. 批量：扩展拉列表 → daemon 编排逐条采集（daemon 加 `kind='collect'` 或复用 job 队列）。

## 八、验收标准

- 在一篇小红书笔记页点"存入飞书表格"，飞书里指定的多维表格新增一条记录：标题/作者/链接
  正确，封面图作为附件能在表里预览；
- 普通网页（如一篇博客）也能存：正文摘要来自 webcopy 转的 Markdown，封面取 og:image；
- 首次建表后，第二次采集复用同一张表、字段不重复建；
- 媒体上传失败有重试和明确报错，不会写出一条"缺图"的坏记录（要么补上要么明确标注）。

## 九、风险与注意

1. **内部接口私有易变**：这是最大风险，也是为什么里程碑 0 是探测。一旦飞书改接口就要跟着修，
   把接口路径/参数集中在 `feishu-bitable.ts` 一处，别散落，便于跟进。
2. **素材上传是最不确定的一环**：内部上传接口可能有大小/类型/频控限制，探测时重点验，拿不到就
   对这一步走 OpenAPI 兜底。
3. **权限边界**：写多维表格需要用户对目标表有编辑权（用登录态天然受飞书权限约束，越权会被接口挡，
   这是对的）。要处理"无权限/表被删"的报错提示。
4. **平台采集的合规**：和视频下载同源——抖音/小红书调内部 API 有风控风险，端点做成可更新配置，
   别硬编码固定发版；第一版优先"用户在页面正常浏览时顺手采集当前这条"，不做大规模爬取。
5. **scope 克制（YAGNI）**：不做社媒助手那种定时任务、cookie 回收上报、云端后台。larksnap 只做
   "用户主动把看到的内容存进自己的飞书表"这一件事。

## 十、工作量估计

里程碑 0 探测约 0.5~1 天（决定成败）；里程碑 1（MVP：通用网页存入指定表）约 2~3 天，其中
bitable 写入和素材上传各约 1 天；里程碑 2（平台适配 + 批量）每个平台约 0.5~1 天、批量编排 1 天，
按需增量。建议先把里程碑 0+1 打通验证登录态内部接口这条路，再决定投入平台适配。
