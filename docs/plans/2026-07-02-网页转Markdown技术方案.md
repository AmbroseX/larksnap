## 1. 架构总览

新功能叫 **webcopy**，和现有飞书导出（feishu 线）完全平行，互不干扰：

```
┌─────────────────────────────────────────────────────────┐
│ Service Worker (background)                              │
│  ├── 现有：feishu 导出 / bridge / cache …（不动）          │
│  └── 新增：webcopy.ts                                     │
│       ├── 注入调度：activeTab 手动注入 / 域名自动注册       │
│       ├── 标签页复制：chrome.tabs.query → 格式化           │
│       ├── 右键菜单：contextMenus 注册与分发                │
│       └── 剪贴板历史：storage 读写（P1）                   │
├─────────────────────────────────────────────────────────┤
│ Content Script（新入口 webcopy.js，与飞书 content 分开）    │
│  ├── index.ts       消息分发、模块挂载/卸载                │
│  ├── unlock.ts      解除复制限制                           │
│  ├── html2md.ts     Readability + Turndown 转换管线        │
│  ├── auto-copy.ts   选中文字自动复制                        │
│  └── float-button.ts 浮动按钮（P1）                        │
├─────────────────────────────────────────────────────────┤
│ UI                                                        │
│  ├── SidePanel：新增 WebCopyView 区块（主入口）             │
│  ├── Options：新增网页复制设置区                            │
│  └── history.html：剪贴板历史页（P1）                       │
└─────────────────────────────────────────────────────────┘
```

设计原则：

- **零侵入默认态**：不开任何开关时，普通网页不注入任何脚本（`content_scripts` 保持空，全部动态注入）
- **单一职责**：webcopy 不碰飞书协议栈；飞书页面默认不挂 webcopy
- **能在 content 做的不进 SW**：DOM 解析、剪贴板写入都在 content 完成，SW 只做调度和跨标签页操作

## 2. 注入与权限流

### 2.1 手动触发（P0，覆盖所有场景）

**前提认知**：`activeTab` 只在四种手势下授予当前标签页——点工具栏图标、点右键菜单项、按快捷键命令、omnibox。**在侧边栏里点按钮不属于其中任何一种**。现有飞书线能随手 `executeScript` 是靠 manifest 里的静态 feishu host 权限兜底，webcopy 面向任意网页没有这个兜底。因此触发路径分两档：

**主路径（右键菜单 / 快捷键，稳定拿到 activeTab）：**

```
用户手势（contextMenus.onClicked / commands.onCommand）
  → SW: chrome.scripting.executeScript({ target: {tabId}, files: ['webcopy.js'] })
  → 脚本幂等挂载（window.__larksnap_webcopy 标记防重复注入；
     ⚠️ 现有 src/content/index.ts 每次注入都重复 addListener 无幂等保护，webcopy 不要学它）
  → SW 发 CONTENT 消息（如 PAGE_TO_MD），content 执行并回包
```

**辅路径（侧边栏按钮）：**

```
按钮点击 → 先尝试 executeScript
  （点图标开侧边栏那一下授予的 activeTab 可能仍覆盖当前标签页，切过标签页则大概率失效）
  → 失败（"Cannot access contents of the page"）
     → 在同一手势内 chrome.permissions.request({ origins: [`*://${host}/*`] })
       （侧边栏页面的用户手势可以发起 permissions.request）
     → 授权成功后重试注入；拒绝则提示改用右键菜单
```

⚠️ 骨架阶段第一件事就是做 15 分钟最小实验（见 §11 第 1 步）验证这两档的实际边界，实验结论回填本节。

### 2.2 域名自动启用（P1）

```
用户在设置页添加域名（用户手势内）
  → chrome.permissions.request({ origins: ['*://*.example.com/*'] })
  → 成功后 chrome.scripting.registerContentScripts({
       id: 'webcopy-auto', matches: [已授权域名...],
       js: ['webcopy.js'], runAt: 'document_idle' })
  → 域名列表变更时 update / 清空时 unregister
```

复用现有 `permissions.ts` 的授权记账思路，但存到独立的 `webcopy.unlockDomains`，不与飞书 `trustedDomains` 混用。

## 3. HTML → Markdown 转换管线（html2md.ts）

### 3.1 依赖选型

| 库 | 用途 | 理由 |
|---|---|---|
| `@mozilla/readability` | 整页正文提取 | Firefox 阅读模式同款，去广告/导航最成熟 |
| `turndown` | HTML → Markdown | 事实标准，规则可扩展 |
| `turndown-plugin-gfm` | GFM 扩展 | 表格、删除线、任务列表 |

不自己手写转换器（SmartCopy 那种 151KiB 手搓方案兼容性差，商店评论里一堆"某网站失效"）。

### 3.2 整页转换流程

```
document.cloneNode(true)                 // Readability 会改 DOM，必须克隆
  → new Readability(clone).parse()       // 得到 { title, content(HTML), byline }
  → 失败降级：取 document.body.innerHTML  // SPA/结构怪异页兜底
  → 预清洗：删 script/style/noscript、隐藏元素（offsetParent 为 null 的跳过）
  → turndown.turndown(html)
  → 拼头部：
      # {title}
      > 来源：{url}　·　抓取时间：{ISO 日期}
  → 返回 { markdown, title }
```

### 3.3 选中内容转换流程

```
window.getSelection().getRangeAt(i).cloneContents()  // 支持多段选区
  → 塞进临时 <div> 序列化为 HTML
  → 相对链接/图片补全为绝对 URL（new URL(href, location.href)）
  → turndown.turndown(html)
```

### 3.4 Turndown 定制规则

- `codeBlockStyle: 'fenced'`、`headingStyle: 'atx'`、`bulletListMarker: '-'`
- 图片：保留绝对 URL 外链（P0 不下载图片——下载/内联 base64 属飞书导出已有能力，后续如需再打通）
- `<pre>` 内嵌套结构（高亮行号表格，如 CSDN/掘金）：自定义规则提纯为纯文本代码块
- 微信公众号 `data-src` 懒加载图：规则里优先读 `data-src`
- 删除规则：`iframe`、`form`、`button` 等交互元素直接丢弃

### 3.5 剪贴板写入策略（按触发来源分流）

关键约束：页面无焦点时（用户手势和焦点都在侧边栏），content 里 `navigator.clipboard.writeText` 会抛 "Document is not focused"，`execCommand('copy')` 同样依赖页面激活态——**不能指望 content 兜一切**。

| 触发来源 | 手势/焦点在哪 | 谁写剪贴板 |
|---|---|---|
| 右键菜单、快捷键、自动复制 | 页面里 | content 直接写（`navigator.clipboard` → 失败降级临时 textarea + `execCommand('copy')`） |
| 侧边栏按钮（整页/选区/标签页复制） | 侧边栏里 | content 只回传 Markdown 字符串，**由侧边栏页面写**（有焦点+手势，稳定可用） |

两条路都失败时回包报错，侧边栏弹出结果预览让用户手动复制。§6 的 COPY_TABS 天然属于第二行模式。

## 4. 解锁模块（unlock.ts）

三层解除，全部可逆（stop() 恢复原状）：

```ts
// ① 事件层：捕获阶段拦截页面的封锁监听器
const EVENTS = ['copy','cut','paste','contextmenu','selectstart','dragstart','mousedown'];
// 对 copy/contextmenu/selectstart 用 stopImmediatePropagation

// ② 样式层：注入 <style data-larksnap-unlock>
//    *, *::before, *::after { user-select: text !important;
//                             -webkit-user-select: text !important; }

// ③ 属性层：清除 document/body 及其祖先上的
//    oncopy/oncut/onpaste/oncontextmenu/onselectstart/ondragstart 内联句柄；
//    MutationObserver 盯着防止页面脚本重新写回
```

不做 `keydown` 拦截：页面脚本先注册的 capture 监听跑在我们前面，`preventDefault` 无法撤销，这条路技术上站不住，还可能误伤正常输入。95% 的禁复制靠 `oncopy/oncontextmenu/selectstart` + CSS，上面三件套已经够用。

已知局限（stopImmediatePropagation 的边界）：若页面在我们注入**之前**就注册了 capture 监听，①层拦不住它——靠 ③层清内联句柄 + ②层 CSS 覆盖补齐；对个别顽固站点，后续可评估 `registerContentScripts` 的 `runAt: 'document_start'`（P1 域名自动启用天然支持）。

## 5. 自动复制模块（auto-copy.ts）

**生效边界（P0）**：自动复制只在"本标签页已手动激活 webcopy（任一入口触发过注入）且开关打开"时生效，**会话级，刷新/关页即失效**。这是"零侵入默认态"的必然推论——没注入过脚本的页面不可能监听选中。真正"打开页面就自动复制"（SuperCopy 域名列表的体验）依赖 P1 的域名自动注册。

```
监听 document 'mouseup' + 'keyup'（防抖 150ms）
  → sel = getSelection()，长度 < minChars(默认5) 则忽略
  → 与上次复制内容相同则忽略（防重复触发）
  → 格式为 markdown 时走 html2md 选区管线，否则 sel.toString()
  → 写剪贴板 → 页面右下角轻提示（2s 自动消失的小 toast）
  → （P1）发 SW 记入剪贴板历史
```

编辑区保护：焦点在 `input/textarea/contenteditable` 内时不触发，避免打字选词误复制。

## 6. 标签页复制（SW 内 webcopy.ts）

```
COPY_TABS { scope: 'current' | 'all', format }
  → chrome.tabs.query({ currentWindow: true, active: scope==='current' })
  → 按 format 拼串：
      markdown  → [title](url)（多条换行）
      title-url → title - url
      title / url → 单字段
  → 结果回传给发起方 UI（侧边栏自身写剪贴板，SW 不碰剪贴板）
```

P2 再加 CSV/JSON/HTML 表格格式（types 里 format 用联合类型，加格式零改动结构）。

## 7. 消息协议与存储

### 7.1 新增 MSG（UI ↔ SW）

| 消息 | data | 返回 |
|---|---|---|
| `WEBCOPY_PAGE_MD` | `{ mode: 'copy'\|'download' }` | `{ markdown, title }`（侧边栏发起时由侧边栏写剪贴板/触发下载，见 §3.5） |
| `WEBCOPY_SELECTION_MD` | — | `{ markdown }`（同上） |
| `WEBCOPY_TOGGLE_UNLOCK` | `{ enabled }` | `{ enabled }` |
| `COPY_TABS` | `{ scope, format }` | `{ text, count }`（侧边栏自身写剪贴板） |
| `CLIP_HISTORY_LIST / DELETE / CLEAR` | `{ id? }` | 列表/成功（P1） |

右键菜单/快捷键路径不经过这些 UI 消息：SW 直接注入后发 CONTENT_MSG，content 在页面手势内自己写剪贴板。

### 7.2 新增 CONTENT_MSG（SW → content）

`WEBCOPY_PAGE_TO_MD` / `WEBCOPY_SELECTION_TO_MD` / `WEBCOPY_UNLOCK` / `WEBCOPY_SET_CONFIG`（配置热更新，改设置不用刷新页面）

### 7.3 存储 schema

```ts
// ExtensionConfig 新增（DEFAULT_CONFIG 同步补默认值）
webcopy: {
  autoCopyEnabled: false,
  autoCopyMinChars: 5,
  autoCopyFormat: 'text',       // 'text' | 'markdown'
  tabCopyFormat: 'markdown',
  unlockDomains: [],            // P1
  floatButton: false,           // P1
  historyLimit: 10,             // P1
}

// 新 STORAGE_KEY（P1）
CLIP_HISTORY: 'feishu2md:clip-history'
// 条目：{ id, type: 'selection'|'page'|'tabs', text(截断5KB), sourceUrl, title, createdAt }
// 写入时按 historyLimit 头部截断；总量兜底上限 200 条
```

## 8. manifest / 构建变更

```diff
 manifest.json
+  "permissions": [..., "contextMenus"]
+  "commands": {                                    // P1
+    "webcopy-page-md":   { "suggested_key": {"default": "Alt+1"} },
+    "webcopy-unlock":    { "suggested_key": {"default": "Alt+2"} },
+    "webcopy-autocopy":  { "suggested_key": {"default": "Alt+3"} },
+    "webcopy-history":   { "suggested_key": {"default": "Alt+4"} }
+  }
```

- **不加 `clipboardWrite`**：content 里 `navigator.clipboard` 靠 transient activation 即可，侧边栏/扩展页写剪贴板同理；该权限主要给扩展页 `execCommand`，真遇到再加
- **host 权限策略（重点）**：不新增任何静态 host 权限。P0 靠右键菜单/快捷键的 `activeTab`；侧边栏兜底和 P1 域名自动启用都走现有 `optional_host_permissions: *://*/*` 之下的运行时 `permissions.request`，与飞书私有化授权同一套底座、不同的记账字段（`webcopy.unlockDomains` vs `trustedDomains`）
- **commands 额度**：Chrome 限制最多 4 个命令带 `suggested_key`，上面正好用满、没有余量；将来若要用 `_execute_action` 会占额度，届时得砍一个

vite.config.ts：新增 `webcopy` 独立入口（IIFE，同现有飞书 content 打法），产物 `dist/webcopy.js`。turndown + readability 只进这个包，飞书导出路径体积不变。

## 9. UI 细节

- **SidePanel**：顶部按当前页类型自适应——飞书文档页显示导出区块（现状），普通网页显示 WebCopyView：〔整页转 Markdown：复制 / 下载〕〔解除复制限制 开关〕〔复制本页链接(MD)〕〔复制全部标签页〕〔自动复制 开关〕
- **右键菜单**：页面右键「整页转 Markdown」「解除复制限制」；选中文本时右键「选中内容转 Markdown」。注册必须放在 `chrome.runtime.onInstalled`（并考虑 `onStartup` 补注册）——MV3 SW 会休眠重启，不能只在模块顶层建一次
- **Options**：网页复制设置卡片（最小字符数、自动复制格式、标签页格式、P1 的域名列表/历史上限/浮动按钮）
- 复制成功统一用页面内 toast 反馈，不用系统通知（少要一个 notifications 权限）

### 与本方案的实现偏差（均为简化，功能等价）

1. **`WEBCOPY_SET_CONFIG` 未实现为消息**：content 直接 `chrome.storage.onChanged` 监听配置变化做热更新，省掉 SW 广播和已注入标签页记账，效果相同。
2. **新增 3 个消息**（§7.1 之外）：
   - `WEBCOPY_GET_STATE`（UI→SW）：只 `tabs.sendMessage` 不注入，无接收方即 mounted=false——侧边栏初始化解锁开关状态时保持零侵入；
   - `WEBCOPY_ENSURE`（UI→SW）：侧边栏打开自动复制开关时注入当前标签页，让开关立即生效；
   - `WEBCOPY_STATE`（SW→content）：查询挂载/解锁状态。
3. **事件层解锁只拦 5 个事件**（copy/cut/paste/contextmenu/selectstart）：§4 EVENTS 列表里的 mousedown/dragstart 用 stopImmediatePropagation 误伤太大（元素级 mousedown 监听全被杀），按 §4 正文"只对 copy/contextmenu/selectstart 系"从严执行。
4. **辅路径的 permissions.request 放在侧边栏拿到 needsPermission 回包后发起**：依赖消息往返仍在 transient activation 窗口内（约 5s），符合 §2.1 设计，但属于最小实验待验证项。
5. **Readability 降级路径未做 offsetParent 隐藏元素过滤**（克隆节点无布局信息拿不到 offsetParent；Readability 主路径自带可见性启发）。

### 站点适配器（2026-07-02 追加，百度文库）

通用管线（Readability+Turndown）对少数站点失效——百度文库正文渲染进 `<canvas>`，
DOM 里只有导航栏文字，真正内容藏在签名的云存储 JSON（`window.pageData.readerInfo.htmlUrls.json`）。
为这类站点加了「站点适配器」机制：

- `src/background/webcopy-adapters.ts`：适配器表 `{ name, match(host), extractor }`；
  `extractor` 通过 `chrome.scripting.executeScript({ world: 'MAIN' })` 在页面主世界执行——
  既能读页面全局变量，fetch 也走页面 origin（跨域 JSON 的 CORS 已放行），且不受页面 CSP 限制。
- 百度文库 extractor：fetch 各页 JSONP（`wenku_N({...})`）→ 取 `body[].c`（逐字 word）→
  按 `ps._enter` 还原换行拼接 → 拼「# 标题 + 来源 URL」头部。
- `webcopyPageMd` 与右键菜单「整页转 Markdown」都先试适配器，命中即用、抓不到退回通用管线；
  右键菜单路径用主世界脚本写剪贴板（页面手势可用）。
- **已知局限**：百度文库免费只放前 N 页（`showPage`/`page`），付费页任何工具都拿不到，
  产出末尾会追加「仅开放前 N/M 页预览」提示。已在真机 DevTools 验证前 2 页正文可完整抓取。
