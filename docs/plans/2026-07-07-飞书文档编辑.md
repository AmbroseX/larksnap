# 飞书文档编辑能力方案

> 目标：在 Claude Code 里给一条飞书文档链接（用户在浏览器里对它有编辑权限），CC 就能把内容写进这篇文档。
> 现状：larksnap 全链路只读（导出 md/pdf/html），没有任何写入能力。

## 一、三条路线对比（先定大方向）

### 路线 A：官方 OpenAPI（lark-cli / lark-doc 技能）

飞书官方的 docx 块编辑接口很完善（创建块、更新块、删除块、Markdown 导入），CC 里已装的
`lark-doc` 技能本身就能编辑文档（`docs +update --api-version v2`）。

- 优点：**零开发**、稳定、官方支持、编辑粒度最细。
- 前提：需要一个企业自建应用（或用户授权），文档所在租户要放行相应 API 权限。
- 结论：**如果你的租户能走通 lark-cli 授权，直接用 lark-doc，本仓库什么都不用改。**
  larksnap 做编辑功能的前提是"这条路走不通"（私有化部署、租户不给开 API、不想找管理员）。

参考实现就在仓库里：`ref/lark-cli`（Go）完整走通了这条路，值得抄的点：

- **身份模型**：OAuth 用户授权拿 **user_access_token**（`/open-apis/authen/v1/authorize`
  浏览器授权 → 本地回调服务收 code → 换 token + refresh）。API 以**用户本人身份**调用，
  所以"用户有编辑权限的链接"就能编辑，不需要逐文档给应用授权。唯一门槛是租户里得有
  一个自建应用（app_id/app_secret）并开通 `docx:document` 等 scopes。
- **读块清单**：`GET /docx/v1/documents/:id/blocks?page_size=500`（分页），返回全量块树
  （block_id / parent_id / block_type / 内容）——就是本方案 `list-blocks` 想要的东西。
- **插入**：`POST /docx/v1/documents/:id/blocks/:block_id/children?document_revision_id=-1`，
  父块 ID 传文档 token 即插到根层级，`index` 控制位置（-1 末尾）。块类型编号：text=2、
  heading1-9=3-11、bullet=12、ordered=13、code=14、todo=17、divider=22、table=31。
- **改块**：`PATCH /docx/v1/documents/:id/blocks/:block_id?document_revision_id=-1`，
  body 是 `update_text_elements`。
- **删块**：ref/lark-cli 没实现，官方有 `blocks/:block_id/children/batch_delete`（按父块
  + 子块下标删），补一个命令即可。
- wiki 链接要先解析：`GET wiki` 节点接口把 node_token 换成 docx token 再操作。

### 路线 B：逆向飞书协同编辑协议（内部接口直写）

larksnap 读路径已经会解 `client_vars` + apool，理论上写路径可以反着来：构造 changeset
提交到协同接口。

- 缺点：飞书编辑是 OT 协同协议（版本号、changeset 编码、冲突合并），逆向工作量极大；
  编码错一个字段就可能**写坏用户文档**；接口无契约保障，飞书前端一升级就崩；风控风险最高。
- 结论：**不做**。读坏了大不了导出失败，写坏了是毁用户数据，风险收益完全不成比例。

### 路线 C：扩展"模拟用户编辑"（推荐）★

复用 larksnap 现有骨架：daemon 推任务 → 扩展开文档标签页 → 在页面里**像用户一样把内容
粘贴进编辑器**。协同保存、版本合并全部由飞书自己的前端代码完成，我们不碰协议。

- 原理：飞书编辑器原生支持粘贴 HTML/Markdown 并自动转成对应的块（标题、列表、代码块、
  表格都认）。我们把要写入的 Markdown 用 `marked`（已有依赖）转成 HTML，合成一个带
  `text/html` + `text/plain` 的 paste 事件投给编辑器，等飞书自动保存完成即可。
- 优点：不逆向写协议，**写坏文档的风险由飞书前端兜底**（它只会产生合法操作）；复用登录态、
  标签页管理、进度回传、错误契约全套现有设施。
- 缺点：编辑粒度粗（见 MVP 范围）；依赖编辑器 DOM 与粘贴行为，飞书前端改版需要跟进；
  有两个必须先验证的技术疑点（见"先做的可行性实验"）。

**结论：选路线 C。** 下文全部围绕路线 C 展开。

> 决策记录（2026-07-07）：确认租户内拿不到自建应用凭证，路线 A 不可用，定路线 C。
> 后文所有"能走通 lark-cli 就用路线 A"的提醒仅作背景保留，不再是备选。
> ref/lark-cli 的价值转为**接口设计参照**：`list-blocks` 的输出结构（block_id / parent_id /
> block_type / 内容摘要的块树）向官方 `GET /docx/v1/.../blocks` 的返回对齐，这样 CC 侧
> 的使用心智和 lark-doc 一致，将来若能切回官方 API 也不用改用法。

## 二、MVP 范围

分两档做：**插入类**（简单，先落地）和**块级修改类**（必做，依赖块定位，紧随其后）。
全部以 Markdown 为输入。

### 2.1 插入类

| 操作 | 说明 | 场景 |
|---|---|---|
| `append` | 在文档**末尾**追加一段 Markdown | 记录追加、日报累积、CC 写结论进文档 |
| `insert-after` | 在**指定标题**（按标题文本匹配）后面插入 | 往文档的某一节里补内容 |

### 2.2 块级修改类（必做）

块级操作都以**块 ID** 为目标，配一个只读的 `list-blocks` 让 CC 先"看清楚再下手"：

| 操作 | 说明 |
|---|---|
| `list-blocks` | 只读。输出文档的块清单：块 ID、类型、层级、内容摘要（前 80 字）。CC 用它定位目标块 |
| `replace-block <id>` | 用新 Markdown **替换**指定块的内容 |
| `delete-block <id>` | 删除指定块 |
| `insert-after-block <id>` | 在指定块后面插入（比按标题文本匹配更精确的插入方式） |

块 ID 的来源与稳定性：

- larksnap 读路径本来就解析块树（`client_vars` / block_map），`list-blocks` 复用它输出，
  **不用新逆向任何接口**；
- 飞书编辑器的 DOM 节点上带块标识（`data-block-id` 一类属性），编辑时用它把
  "清单里的块 ID"对应到"页面上的 DOM 节点"——这条映射是否处处成立要进实验验证（实验 3）；
- 块 ID 在编辑之间是稳定的（协同协议的锚点就是它），但**别人同时在编辑时**目标块可能
  被改/被删：操作前校验"该块当前内容摘要"与 `list-blocks` 时一致，不一致就报错重取，
  宁可失败不可改错。

明确不做（真二期）：表格单元格级编辑、评论、多人光标处插入、格式刷类样式微调。

> 提醒：块级修改正是官方 OpenAPI 的强项（原生按 block_id 增删改，有契约保障）。如果
> lark-cli 授权走得通，路线 A 做块级编辑比路线 C 可靠得多——启动开发前值得再试一次。

编辑前置校验：打开文档后检测页面是否可编辑（只读视图没有编辑器实例 / 有"只读"标识），
不可编辑回新错误 `need-edit-permission`，提示用户去浏览器确认权限。

写入后校验：复用现有 Markdown 导出引擎**回读全文**，确认追加的内容出现在导出结果里，
校验通过才回成功；不通过按导出失败契约报错（内容可能已进文档，提示用户人工确认）。

## 三、先做的可行性实验（动手写功能之前必须过的三关）

这三个疑点不解决，后面的设计都是空谈。各花半天，用 feishu-doc-probe / Playwright 在真实
文档上手工验证：

**实验 1：合成 paste 事件飞书编辑器认不认（isTrusted 问题）**

脚本合成的 `ClipboardEvent('paste')` 是 `isTrusted: false`，飞书编辑器可能直接忽略。
按优先级逐个试，取第一个能用的：

1. 主世界注入脚本，对编辑器根节点 dispatch 带 `DataTransfer`（`text/html` + `text/plain`）
   的 paste 事件 —— 成本最低，先试这个。
2. `document.execCommand('insertHTML' / 'insertText')` —— 老 API 但很多自研编辑器认。
3. `chrome.debugger` 附加标签页，用 CDP 的 `Input.insertText` / 按键序列 —— 100% 是可信
   输入，一定能进编辑器，代价是页面顶部会出现"正在调试此浏览器"横幅（后台标签页用户
   基本看不到，可接受，但要写进文档说明）。

**实验 2：后台标签页（active:false）里编辑器是否正常工作**

导出只要拿数据，后台标签页够用；但编辑器可能懒加载、光标/选区在不可见页面可能失效、
自动保存可能被降频。验证：后台开标签页 → 注入 → 追加 → 回读。如果不行，降级方案是
`active: true` 短暂前台（体验差一点但能用），或 `chrome.windows.create({ focused: false })`
开独立小窗口。

**实验 3：块级选区能不能程序化操纵（块修改类操作的命门）**

`replace-block` / `delete-block` 都要求"精确选中某个块的全部内容再替换/删除"。要验证：

1. 块清单里的块 ID 能否稳定对应到页面 DOM 节点（找 `data-block-id` 一类属性，
   docx / wiki 两种壳都要看）；
2. 用 `Selection.setBaseAndExtent`（或飞书自己的选区模型）把选区精确铺满一个块后，
   投 paste / Delete 键，飞书是替换整块还是行为异常（拆块、残留、把相邻块卷进来）；
3. 兜底路径：CDP 鼠标点进块首 → `Shift+Cmd/Ctrl+End` 或块级快捷键选中 → 键入删除。
   验证嵌套块（列表项、引用块、代码块）不会误伤兄弟块。

三关都过 → 按下文实施；任何一关 3 天内过不了 → 回到路线 A，把结论写进本文档后归档。
（实验 1、2 挡的是全部功能，实验 3 只挡块修改类：如果只有实验 3 失败，插入类照常做，
块修改类降级为"路线 A 或人工"，结论同样写回本文档。）

### 实验结论（2026-07-07 实测回写，iflytek 私有化租户真实文档验证）

**实验 1 —— 合成事件全军覆没，CDP 可信输入是唯一通路：**

- 合成 paste、合成 beforeinput（insertFromPaste）、`execCommand('insertHTML'/'insertText')`
  全部被飞书编辑器忽略。飞书只认 `isTrusted: true` 的输入。
- 一个差点酿祸的坑：最早用 MutationObserver"有 DOM 突变"判定注入是否生效，结果
  focus/选区变化触发的编辑器自渲染造成**假成功**。判定必须用"内容指纹"——注入后编辑器
  `innerText` 里真的出现内容才算数；且指纹要用**剥掉 Markdown 语法的首行 + 末行**
  （innerText 是渲染结果，带 `#`、`-` 的原文永远匹配不上，这个错误曾导致误判失败后
  重试、重复写入三份；长文档飞书虚拟化渲染，粘贴后视口在尾部，首行可能不在 DOM 里，
  末行指纹兜底）。校验超时按内容大小自适应（19KB 实测 5s 不够，渲染要 20s+）。
- 写剪贴板**绝不能信任 API 返回值**：`execCommand('copy')` 在后台页会返回 true 但实际
  没写进系统剪贴板（曾把上一次任务的旧剪贴板内容贴进文档造成污染）。最终方案：
  async Clipboard API 写入 + 隐藏 textarea 执行 paste **回读比对**，校验不过重试三次，
  仍不过就中止——宁可失败也不动文档。
- 粘贴口味的选择（重要）：HTML 口味粘贴会把表格转成**内嵌电子表格（sheet 块）**，
  其单元格数据靠页面组件异步提交，后台页+虚拟化渲染下经常提交不了，留下永远转圈的
  空表格；代码块里 `#` 开头的行还会被不稳定地解析成标题。**默认改用纯文本 Markdown
  口味**（只给 text/plain，飞书自己的 md 粘贴解析转块）：表格转成原生简单表格
  （数据直接在文档模型里）、代码块注释行保真、嵌套列表/行内格式全部正常。
  `--html-paste` 保留为退路开关。
- 权限代价：`debugger` 是 Chrome 明文规定**不能作为可选权限**的（试过 optional_permissions
  + 弹窗授权，request 直接失败），只能进必选 permissions，商店升级时用户会看到调试权限
  警告；同时需要 `clipboardRead`（textarea 回读校验 + execCommand('paste') 可信粘贴）。
- 副作用：编辑期间占用系统剪贴板；任务执行几秒内页面顶部有"正在调试此浏览器"横幅。

**实验 2 —— 后台标签页可用，但要焦点仿真：**

- `active:false` 后台标签页里编辑器正常实例化（visibility=hidden 不影响）。
- 关键前提：`Emulation.setFocusEmulationEnabled` 开焦点仿真，否则 Blink 的编辑命令
  （Copy 等）对非激活页面不执行——这是第一版 Copy 静默失败的根因。

**实验 3 —— DOM 选区驱动不了飞书，靠可信鼠标点击 + Esc 块选中：**

- 块 ID → DOM 节点：`data-record-id` 属性稳定命中（docx，私有化租户实测）。
- `Selection.setBaseAndExtent`/Range 铺满整块后按退格，飞书把选区归一成块尾光标，
  **只删了一个字**——DOM 选区不进飞书自己的选区模型。
- 可行路径：可信鼠标点击块中心（飞书自己定位光标）→ `Esc` 进入块选中态 → Backspace
  删整块 / 粘贴替换整块。插入类则点击锚点块文本末端（Range 行盒过滤零宽矩形取最靠下的，
  否则光标会飘到块首）→ **先回车新起一块**再粘贴（否则粘贴首段并进锚点块内部）。
- 两个衍生事实：`replace-block` 成功后**块 ID 会变**（飞书按删旧插新处理），后续操作必须
  重新 list-blocks；replace/delete 的生效判据用"旧块 DOM 节点消失"（文本长短判据会被
  "只删一个字"或"换入更短内容"骗过）。

**实验 4 —— 长文档虚拟化渲染会骗过所有"看 DOM"的判据（2026-07-08 翻案实测）：**

- 现象：写入 20KB/32KB 文档后，`list-blocks` 永远只报 **238 块**，回读校验尾指纹匹配
  失败报 `save_unconfirmed`，连 `fetch` 的 md 导出都只有 19KB——三个"仪器"一致指向
  "内容截断"。曾据此怀疑服务端有块数配额，折腾了分片粘贴、WS 静默等待、页面反节流，
  全部无效（截断位置纹丝不动）。
- 真相：**内容从头到尾都写入成功了**（人工滚到文档底部可见全部内容）。飞书长文档
  虚拟化渲染，DOM 里最多只挂约 238 块（固定窗口），`list-blocks`、内容指纹校验、
  md 导出全是从 DOM 取数，看到的都是渲染窗口而不是文档全貌。两次"恰好 238"是窗口
  容量，不是服务端截断。教训：**判据仪器本身要先校准——用服务端/人工视角交叉验证，
  别让三个同源（都看 DOM）的证据互相"确认"**。
- 连带故障：`append` 定位"最后一个块"也是看 DOM，长文档下 DOM 末块 ≠ 文档真末块，
  追加内容会**落到文档中间**（实测 5 行测试内容插进了正文中部）。长文档（约 238 块
  以上）目前四处都不可靠：list-blocks 不全、append 定位错、回读校验误报、md 导出不全。

**验收记录（2026-07-07）：** 六个子命令全过（append / insert-after / insert-after-block /
replace-block / delete-block / list-blocks），批量删除 12 块无失手；错误分支 anchor_not_found /
block_not_found / block_changed / need_edit_grant(退出码 7) 按契约报错；四步演示（添加→追加→
编辑→删除）与 19KB 本计划文档整篇写入均成功（107 块落地）；md-paste 模式下标题/加粗/列表/
嵌套列表/引用/代码块/表格全要素保真。公有云文档验收待做。

## 四、技术方案（路线 C 细化）

### 4.1 总体流程

```
CC ── edit.mjs ──HTTP /command──▶ daemon ──WS──▶ 扩展
                                                  │ 1. 校验域名授权（复用 permissions）
                                                  │ 2. 后台开文档标签页，等加载完成
                                                  │ 3. 检测登录态 / 可编辑（否则 need-*）
                                                  │ 4. 定位插入点（文末 / 目标标题后）
                                                  │ 5. marked 把 Markdown 转 HTML，注入编辑器
                                                  │ 6. 等飞书自动保存完成（同步状态标识）
                                                  │ 7. 复用导出引擎回读，校验内容已落地
                                                  └─▶ result / error 经 WS 回传 CC
```

### 4.2 各层改动

| 层 | 文件 | 改动 |
|---|---|---|
| 桥接协议 | `skills/larksnap-fetch/scripts/bridge/protocol.mjs` | 协议版本 +1；job 增加 `kind: 'edit'`、`op`、`contentMd`、`anchor` 字段 |
| daemon | `scripts/bridge/daemon.mjs` | 透传新字段即可（daemon 不理解语义，无逻辑改动） |
| CLI | 新增 `scripts/edit.mjs` | 子命令式用法（见 4.3）；写入内容一律走 md 文件不走命令行参数（避免转义和长度问题）；`list-blocks` 结果以 JSON 打到 stdout；错误契约与 fetch.mjs 完全一致，新增退出码 `6 = need-edit-permission` |
| 扩展 SW | `src/background/bridge.ts` | `onMessage` 分发 `edit` 任务 → 新增 `src/background/editor.ts` 编排（开标签页 → 注入 → 等保存 → 回读校验）；`list-blocks` 走现有读管线，不开编辑流程 |
| content | 新增 `src/content/editor/` | 编辑器探测（可编辑判定）、块 ID→DOM 定位、插入点/块选区定位、Markdown→HTML→注入（按实验 1/3 的结论实现）、保存状态监听 |
| Skill | `skills/larksnap-fetch/SKILL.md` | 增加"编辑文档"用法与错误分支说明（技能名不变，编辑是 fetch 的姊妹动作，共用 daemon） |

### 4.3 关键实现点

- **CLI 子命令**（目标块的定位统一先 `list-blocks` 再操作）：

  ```bash
  node edit.mjs <链接> append <md文件>
  node edit.mjs <链接> insert-after "<标题文本>" <md文件>
  node edit.mjs <链接> list-blocks                      # 只读，输出块清单 JSON
  node edit.mjs <链接> replace-block <块ID> <md文件>
  node edit.mjs <链接> delete-block <块ID>
  node edit.mjs <链接> insert-after-block <块ID> <md文件>
  ```

- **插入点定位**：`append` = 把选区移到最后一个块末尾；`insert-after` = 遍历标题块按
  文本精确匹配（匹配不到 / 匹配到多个都报错，宁可失败不可插错地方）。
- **块级操作的防呆**：`replace-block` / `delete-block` 下发时随任务带上 `list-blocks`
  当时看到的**内容摘要**，扩展执行前先比对目标块当前内容——对不上说明文档在这期间被
  改过，报 `block_changed` 让 CC 重新 `list-blocks`。删除嵌套容器块（列表、引用等）时
  连同其子块一起删，并在 `list-blocks` 清单里把父子关系标清楚，避免 CC 误删一整节。
- **保存确认**：飞书编辑器有同步状态（页面里"保存中/已保存"标识或内部同步队列）。
  监听它，超时 15s 视为保存不确定 → 走回读校验兜底。
- **回读校验**：取追加内容的首行/末行做包含性检查即可（导出转换与粘贴解析对复杂语法
  的归一化可能不完全一致，不做全文逐字对比）。
- **安全边界**：编辑任务只对已授权域名开放（复用现有 permissions 白名单）；daemon 的
  HMAC 签名机制原样覆盖新命令，本机其他进程无法伪造编辑请求。

### 4.4 错误契约扩展

沿用"stderr 最后一行 JSON"契约，新增：

| 退出码 | type | subtype | 含义 |
|---|---|---|---|
| 6 | authorization | `need_edit_permission` | 已登录但对该文档只读 |
| 1 | edit | `anchor_not_found` / `anchor_ambiguous` | `insert-after` 找不到 / 匹配到多个标题 |
| 1 | edit | `block_not_found` | 块 ID 在当前文档里不存在（已被删或 ID 传错） |
| 1 | edit | `block_changed` | 目标块内容与 `list-blocks` 时不一致，需重取清单 |
| 1 | edit | `save_unconfirmed` | 已注入但保存与回读校验都未确认（提示人工检查文档） |
