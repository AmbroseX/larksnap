# doctor 自检命令 + 扩展日志上报

> 状态：待人工审查（已按 2026-07-09 外部审查意见修订：diag 的 url 守卫、client.mjs 复用实情、
> console hook 幂等、协议横幅副作用、uptime 时间戳、Node 判据、401/403 区分、删掉 DELETE /logs）
> 来源：对照 ref/OpenCLI 的功能盘点，选定「doctor + 日志缓冲」作为第一个移植包。
> 借鉴对象：OpenCLI `src/doctor.ts`（自检）、`src/daemon.ts` 的 `/logs` 环形缓冲、`extension/src/background.ts` 的 console 转发。

## 1. 背景与目标

larksnap 现在排查问题只有两条路：翻 `~/.larksnap/daemon.log`，或让用户打开扩展的
Service Worker DevTools 看控制台。用户报「用不了」时，我们没有一个命令能一次性回答：

- daemon 起了没有、版本对不对？
- 扩展连上没有、哪个 profile、扩展版本和协议版本匹配吗？
- 任务派发链路真的通吗（不只是 WS 握手成功）？
- 编辑功能需要的 debugger 权限授了没有？
- 最近扩展侧报过什么错？

目标：加一个 `doctor.mjs` 自检命令，一条命令跑完上面所有检查，输出人能读的分区报告；
同时让扩展侧的 console 日志流到 daemon 的内存环形缓冲，doctor（和后续排查）能直接取到，
不再需要用户开 DevTools。

## 2. 现状梳理（改动落点）

| 组件 | 文件 | 现状 |

## 3. 方案设计

### 3.1 doctor 命令（新文件 `skills/larksnap-fetch/scripts/doctor.mjs`）

用法：`node doctor.mjs [--profile <code>] [--json]`

检查流程（借鉴 OpenCLI「先跑一次真实探测、顺带把 daemon 拉起来」的思路）：

1. **环境**：Node 版本（**判据：≥18 为 `[OK]`，低于 18 为 `[WARN]` 并提示升级**——脚本只用 `node:` 内置模块，18 取的是保守的 LTS 下限）；技能文件完整性（`bridge/daemon.mjs`、`client.mjs`、`protocol.mjs` 是否都在）。
2. **daemon**：`ping()` 探活；没起就走现成的 `ensureDaemon()` 拉起（版本漂移自愈也顺带完成）。
   报告 daemon 版本、pid（读 `~/.larksnap/daemon.pid`）、端口（含 `LARKSNAP_PORT` 覆盖提示）。
3. **扩展连接**：`GET /status`（见 3.3 扩充后的返回）。
   - 0 个 profile → `[MISSING]`，给三步指引（打开 Chrome → 确认扩展已加载 → 点图标唤醒）。
   - 多个 profile 且未指定 `--profile` → 全部列出，提示如何指定；不视为失败。
   - 每个 profile 打印：code、扩展版本、协议版本、连上时长。
4. **版本兼容**：扩展 `protocolVersion` 与 daemon `PROTOCOL_VERSION` 比对，不一致 → `[WARN]` 提示更新扩展或技能。
5. **端到端探测**：向目标 profile 发一个 `kind:'diag'` 的轻量任务（见 3.2），拿回扩展运行时信息：
   - 扩展版本（manifest 实时值）
   - debugger 权限是否已授予（编辑功能的前提，`chrome.permissions.contains`）
   - 已授权域名数量（`LIST_TRUSTED` 现成逻辑）
   - 往返耗时
   扩展协议版本 < 3（旧扩展）时跳过此项并 `[WARN]`：daemon 侧按 `_proto` 挡在派发前（同现有 edit 的处理，回 `extension_outdated`）。
   注意挡的理由是**让 CLI 拿到明确错误而不是静默超时**——旧扩展的接收守卫（见 3.2）本来就会把无 url 的消息直接丢掉，并不会误当抓取任务执行。
6. **最近日志**：`GET /logs` 取环形缓冲（见 3.4），只打印最近 20 条 warn/error 级别的，全绿时不打印。

输出格式（纯文本分区 + `[OK]/[WARN]/[MISSING]/[FAIL]` 标记，仿 OpenCLI）：

```
larksnap doctor

环境
  [OK] Node v22.1.0
  [OK] 技能文件完整

daemon
  [OK] 运行中 v1.4.0 (pid 12345, 127.0.0.1:19925)

扩展
  [OK] 1 个 profile 已连接
       - k3f9mh2a  扩展 v0.4.0  协议 v3  已连 2h
  [OK] 协议版本匹配 (v3)

连通性
  [OK] 端到端探测通过 (48ms)
  [OK] 编辑权限（debugger）已授予
  [OK] 已授权 2 个私有化域名

一切正常！
```

退出码：无 `[FAIL]`/`[MISSING]` → 0，否则 1（doctor 是诊断工具，不复用 fetch/edit 的
subtype 派生退出码；`--json` 时 stdout 输出完整结构体，供 AI 消费）。

`--json` 结构（示意）：

```json
{
  "ok": true,
  "daemon": { "running": true, "version": "1.4.0", "pid": 12345, "port": 19925 },
  "profiles": [{ "contextId": "k3f9mh2a", "extensionVersion": "0.4.0", "protocolVersion": 3, "connectedAt": 1751940000000 }],
  "connectivity": { "ok": true, "elapsedMs": 48, "debuggerGranted": true, "trustedDomains": 2 },
  "issues": []
}
```

### 3.2 diag 探测任务（协议 v3）

扩展侧 `bridge.ts` 在 `runJob` 之前拦截 `kind:'diag'`，**不开标签页、不进导出引擎**，
直接收集信息回 `result`：

```
daemon ──WS { type:'job', id, kind:'diag' }──▶ 扩展
扩展  ──WS { type:'result', id, diag:{ extensionVersion, protocolVersion,
             debuggerGranted, trustedDomains } }──▶ daemon ──▶ CLI
```

- `extensionVersion`：`chrome.runtime.getManifest().version`（实时，不用常量）。
- `debuggerGranted`：`chrome.permissions.contains({ permissions: ['debugger'] })`。
- `trustedDomains`：复用现有 LIST_TRUSTED 的统计逻辑，只回数量不回域名列表（诊断输出不带用户环境细节，遵循宪法的脱敏原则）。
- daemon 对 `_proto < 3` 的连接不派发 diag（复用现有 edit 的挡法，`daemon.mjs:181` 同款），
  CLI 收到 `extension_outdated` 时降级为 `[WARN] 扩展版本旧，跳过端到端探测`。
- `PROTOCOL_VERSION` 2 → 3；`DAEMON_VERSION` 1.3.0 → 1.4.0。
- **diag 没有 `url`，两端的 url 校验都要放行**（不改就是「发了 diag 什么都不发生」的无症状 bug）：
  - daemon `/command` 现有的 `if (!job.url)` 校验放行 `kind:'diag'`；派发时消息里没有 url 键
    （`JSON.stringify` 会把 undefined 键丢掉，属预期，别写成 `url: ''` 之类的占位）。
  - **扩展侧 `bridge.ts:222` 的接收守卫是 `if (msg.type === 'job' && msg.id && msg.url)`，
    会在进入 kind 判断之前就把无 url 的 diag 丢掉**。必须改成分支：`kind:'diag'` 只要求
    `type + id`；抓取/编辑任务照旧要求 `url`。这是本方案最容易漏的一刀。

### 3.3 /status 扩充

现在 `/status` 只回 `profiles: ["k3f9mh2a"]`（code 字符串数组）。改为对象数组：

```json
{ "ok": true, "daemonVersion": "1.4.0", "pid": 12345, "uptimeSec": 7200,
  "profiles": [{ "contextId": "k3f9mh2a", "extensionVersion": "0.4.0",
                 "protocolVersion": 3, "connectedAt": 1751940000000 }],
  "pending": 0 }
```

配套改动：
- daemon 在 hello 时把 `msg.version` 存到 `conn._extVersion`（现在丢掉了），连接时间存 `conn._connectedAt`。
- daemon 现在没有记启动时间，`uptimeSec` 要新增一个模块级启动时间戳变量来算（trivial，但别漏）。
- 兼容性：搜了代码，当前没有任何 CLI 消费 `/status`（client.mjs 只用 `/ping`），
  且 CLI 与 daemon 同技能目录分发、有版本自愈，直接改结构无风险。

### 3.4 日志环形缓冲 + 扩展 console 上报

**daemon 侧（daemon.mjs）**：
- 内存环形缓冲 200 条，条目 `{ ts, source, level, text }`，`source` 为 `'daemon'` 或 `'ext:<contextId>'`。
- daemon 自己的 `log()` 同时进缓冲（level 固定 `info`），保持写 `daemon.log` 文件不变。
- 新增 WS 消息类型 `{ type:'log', level, text }`（扩展 → daemon），入缓冲，不写文件（避免刷爆 daemon.log）。
- 新端点 `GET /logs`（回 JSON 数组），走现有 HMAC 鉴权（非 `/ping` 端点默认要求，无额外代码）。
  不做 `DELETE /logs`（YAGNI：环形缓冲自己滚动覆盖，doctor 用不上清空，真要了再加）。
- 单条 text 截断 500 字符，防止大对象日志撑爆内存。

**扩展侧（bridge.ts）**：
- hook 全局 `console.log/warn/error`（仿 OpenCLI）：原样调用原函数，同时把参数
  拼成一行文本经 WS 发 `{ type:'log', ... }`。
- **hook 只装一次**：安装点放模块加载或 `startBridge()` 里，**不能放 `connect()` 的 onopen 里**——
  `connect()` 会被 alarms 保活（~24s）和断线重连反复触发，放那儿 hook 会层层套娃、日志成倍复制。
  发送时动态引用「当前 ws」变量即可。
- 防递归：hook 内部发送失败时静默丢弃，绝不再走 console。
- WS 未连接时直接丢弃（不做本地缓存补发，YAGNI；SW 唤醒后的新日志自然会流过去）。
- 隐私边界：日志只进本机 daemon 的内存缓冲（127.0.0.1，HMAC 鉴权后才可读），不落盘、不出机器，
  与宪法「内容不外发」一致。日志里可能含文档 URL/标题，属本机诊断可接受范围，SKILL.md 安全段落补一句说明。

**CLI 侧**：
- doctor 集成（见 3.1 第 6 步）。
- `client.mjs` 的 `export_failed` 等 hint 从「查看 ~/.larksnap/daemon.log」升级为
  「跑 `node <skill>/scripts/doctor.mjs` 自检」。
- 不做独立 `logs` 子命令（KISS，doctor 够用；真需要时 `curl` /logs 也行）。

### 3.5 附带小修：EXTENSION_VERSION 常量陈旧

`src/shared/constants.ts:4` 的 `EXTENSION_VERSION='0.1.0'` 与 manifest 的 0.4.0 脱节，
诊断报告（`src/background/diagnostic.ts:71`）用的就是它。修法：删除该常量，所有引用处改为
`chrome.runtime.getManifest().version`（运行时取，永不再漂移）。

## 5. 边界（这次不做）

- **不做**独立 `logs` / `daemon restart` 子命令（doctor + 现有版本自愈已覆盖）。
- **不做** profile 别名（`profile rename/use`）——单列一个后续计划。
- **不做**扩展侧日志的本地缓存补发（WS 断开期间日志直接丢）。
- **不改**现有退出码契约与 fetch/edit 的任何行为。
- **不动** OpenCLI 的 sysexits 退出码风格——larksnap 已有自己的契约。

## 6. 兼容性与风险

| 风险 | 应对 |
|---|---|
| 旧扩展（proto≤2）收到 diag | 不会误当抓取任务——无 url 的消息在旧扩展也被 222 行守卫直接丢掉。daemon 仍按 `_proto < 3` 挡在派发前（同 edit 路径），目的是让 CLI 拿到明确的 `extension_outdated` 而不是等超时 |
| PROTOCOL_VERSION 2→3 后，所有没重新构建加载的旧扩展 popup 会弹「协议版本不一致」横幅（`Popup.tsx:105`） | **接受**。横幅语义正确（确实该更新了），fetch/edit 功能本身不受影响，用户按提示重建扩展即消。发版说明里写一句 |
| 旧 daemon（1.3.0）不认识 `/logs`、diag | doctor 起手用 `ensureDaemon()`，版本自愈保证 daemon 必是 1.4.0，无混跑窗口 |
| `/status` 结构变化 | 无现存消费方（client.mjs 只用 /ping），CLI 与 daemon 同目录分发 |
| console hook 影响扩展正常日志 | 原函数照常调用，上报失败静默；DevTools 里看到的日志不变 |
| 日志含文档 URL/标题 | 只进本机内存缓冲、HMAC 鉴权后可读、不落盘；SKILL.md 说明 |

## 7. 验收清单（实测）

1. daemon 未启动时跑 `doctor.mjs`：自动拉起 daemon，全绿输出，退出码 0。
2. Chrome 未开（或扩展未加载）时跑：扩展区 `[MISSING]` + 三步指引，退出码 1。
3. 两个 Chrome profile 都连上时跑：列出两个 profile；`--profile <code>` 指定后连通性探测只打指定的那个。
4. 未授予 debugger 权限的浏览器：连通性区 `[WARN]/[MISSING]` 提示编辑功能不可用，抓取不受影响。
5. 手动把扩展里 `PROTOCOL_VERSION` 改回 2 重载：doctor 报 `[WARN]` 协议版本不匹配 + 跳过端到端探测。
6. 在扩展 SW 里 `console.error('test-marker')`，跑 `doctor.mjs`：最近日志区出现该条（source 为 `ext:<code>`）。
7. `curl` 访问 `/logs` 的鉴权：不带签名头 → **403**（missing auth header，`daemon.mjs:116`）；
   带头但签名错误 → **401**（`daemon.mjs:142`）。两种都测，别混为一谈。
8. 跑一次真实 `fetch.mjs` 抓取 + 一次 `edit.mjs append`：行为与改动前一致（回归）。
9. 扩展诊断报告（EXPORT_DIAGNOSTIC）里版本号显示 manifest 实际版本（0.4.x 而非 0.1.0）。
