# 本地 Markdown 预览与编辑

> 日期：2026-07-15
> 状态：待人工审查，审查通过后再编码

## 一、要解决的问题（第一性原理）

用户想在扩展里打开本地磁盘上的 `.md` 文件，一边改源码一边看渲染效果，改完能存回**原来那个文件**。

拆成三件最核心的事：

1. **打开**：从磁盘选一个 `.md` 文件，读出文本。
2. **看**：左边写源码，右边实时渲染，所见即所得。
3. **存**：改完直接写回原文件（不是另存新文件）。

## 二、关键决策（已和用户确认）

| 决策点 | 选择 | 理由 |
| --- | --- | --- |
| 界面放哪 | 独立整页 Tab（新建 `editor.html`） | 侧边栏只有 ~360px，编辑长文太挤；整页才放得下左右分栏 |
| 怎么存 | 写回原文件（File System Access API） | 拿到可写句柄，真正原地保存，这才是「编辑本地文件」 |
| 怎么排布 | 左右分栏，右侧实时预览 | 所见即所得，长文写作最顺手 |

## 三、为什么不需要动 background / 不需要新权限

- `editor.html` 是扩展自己的页面（和 options/sidepanel 同级），里面能直接用浏览器全部 DOM 能力。
- **File System Access API**（`showOpenFilePicker` / `handle.createWritable`）在扩展页面里可直接调用，**不需要在 manifest 里加任何权限**，也不用经过 background 转发。
- 所以这个功能是**纯前端、自包含**的一页，不碰现有的飞书/网页抓取管线，风险隔离得很干净。

## 四、能复用的现成东西

| 复用 | 位置 | 说明 |
| --- | --- | --- |
| Markdown 渲染 | `src/sidepanel/Markdown.tsx` | marked → sanitize 白名单 → 代码高亮，直接拿来当右侧预览 |
| HTML 消毒 | `src/sidepanel/sanitize.ts` | Markdown.tsx 内部已用，无需额外处理 |
| 多语言 | `src/shared/i18n/useI18n` | 新界面文案走同一套 i18n |
| 多页构建 | `vite.config.ts` 的 `rollupOptions.input` | 照 options/sidepanel 的样子加一个 `editor` 入口 |

> 注意：`Markdown.tsx` 目前放在 `src/sidepanel/` 下。它已经是通用组件（不依赖侧边栏状态），编辑器页要引用它。为避免「编辑器依赖侧边栏目录」这种别扭的依赖方向，计划把 `Markdown.tsx` + `sanitize.ts` 挪到 `src/shared/`（或新建 `src/shared/markdown/`），两处都从新位置引用。这是纯移动 + 改 import，不改逻辑。

## 五、界面设计

整页布局（`editor.html`）：

```
┌───────────────────────────────────────────────────────────┐
│ 工具条：[打开]  文件名.md ●(未保存圆点)   [另存为] [保存 ⌘S] │
├──────────────────────────────┬────────────────────────────┤
│                              │                            │
│   左：源码编辑区              │   右：实时渲染预览          │
│   （textarea）               │   （复用 Markdown 组件）    │
│                              │                            │
│   滚动 ── 按比例同步 ──────────▶ 滚动                       │
│                              │                            │
└──────────────────────────────┴────────────────────────────┘
```

- **左侧编辑**：用原生 `<textarea>`。KISS——不引入 CodeMirror/Monaco 这类重依赖（项目一贯克制依赖），先把「能改能存能预览」做扎实。`Tab` 键插入缩进、`⌘/Ctrl+S` 保存拦截，这些用少量代码补齐。
- **右侧预览**：直接 `<Markdown text={内容} />`。输入时用防抖（~150ms）触发重渲，长文不卡。
- **顶部工具条**：打开、文件名、未保存圆点、保存、另存为。
- **未保存保护**：内容有改动且未保存时，`beforeunload` 弹浏览器原生「确定离开？」提醒。

## 六、文件读写流程（File System Access API）

**打开**：
1. 用户点「打开」→ `window.showOpenFilePicker({ types: [{ description: 'Markdown', accept: { 'text/markdown': ['.md', '.markdown'] } }] })`（必须由点击手势触发）。
2. 拿到 `FileSystemFileHandle` → `handle.getFile()` → `file.text()` 得到文本，填进 textarea。
3. 句柄存进组件 state（用于保存）。

**保存（⌘S / 点保存）**：
1. 若还没有句柄（新建的空白内容）→ 走「另存为」。
2. 有句柄 → 先 `handle.queryPermission({ mode: 'writable' })`，非 `granted` 就 `handle.requestPermission({ mode: 'writable' })`（首次会弹一次授权，之后同一会话免再问）。
3. `const w = await handle.createWritable(); await w.write(内容); await w.close();`
4. 成功后清掉「未保存圆点」，工具条给个「已保存 HH:MM」提示。

**另存为**：`window.showSaveFilePicker({ suggestedName, types })` → 拿新句柄 → 写入 → 之后保存都指向新句柄。

**兜底**：进页面先探测 `'showOpenFilePicker' in window`。理论上 Chrome 扩展页都支持；万一不支持，工具条禁用并提示「当前环境不支持本地文件读写」。

## 七、入口（怎么打开这个编辑器）

- 在侧边栏工具区（`src/sidepanel/ToolGroups.tsx`）加一个按钮「本地 Markdown 编辑器」，点了执行
  `chrome.tabs.create({ url: chrome.runtime.getURL('editor.html') })`，在新标签页打开整页编辑器。
- 先只做这一个入口（YAGNI）。右键菜单 / 快捷键 / 关联 .md 文件默认打开等，留作后续增量。

## 九、验收标准

1. 侧边栏点「本地 Markdown 编辑器」→ 新标签页打开整页编辑器。
2. 点「打开」选一个磁盘上的 `.md` → 左侧显示源码、右侧显示渲染结果。
3. 左侧改字 → 右侧实时更新；左侧滚动 → 右侧按比例跟随。
4. `⌘/Ctrl+S` → 内容写回**原文件**（用文件管理器/其他编辑器打开确认已改）；工具条显示已保存。
5. 有未保存改动时关标签页 → 浏览器弹「确定离开？」。
6. 「另存为」→ 选新路径存出新文件，之后保存指向新文件。
7. 中英文切换，编辑器界面文案跟随。

## 十、明确不做（本期 YAGNI）

## 追记（2026-07-15 首轮实测后修正）

真机测试暴露两个问题，已修：

### 1. 受管浏览器里 `showOpenFilePicker` 不存在 → 别死在检测上
原设计不支持就禁用工具条。实测有的环境（疑似受管企业浏览器/内核差异）
`window.showOpenFilePicker` 直接缺失。改为**优雅降级不断链**：
- 有 File System Access：打开文件走它，拿可写句柄，`⌘S` 写回原文件。
- 没有：打开文件退回 `<input type=file>`（只读），保存按钮变「下载」，存为新文件副本。

### 2. 相对路径图片不显示（`![](images/xxx.png)`）
两个原因：① 共享 `sanitize` 为对话页隐私**默认删所有图片**；
② 编辑器页是 `chrome-extension://` 源，`images/xxx.png` 解析到扩展目录（404），
单开一个文件也够不到旁边的 `images/` 目录。修法：
- `sanitize` 加 `allowImages` 开关（默认 false，对话页不受影响；编辑器传 true 保留图片，
  src 仅放行 http/https/data/blob，`javascript:` 等危险 src 拦掉，`on*` 不在白名单）。
- `Markdown` 加 `resolveSrc`：把相对路径换成 blob URL。
- 编辑器新增**「打开文件夹」**（`<input webkitdirectory>`，任何环境可用）：整目录读进内存，
  给所有图片建 `相对路径 → blob` 映射，预览时相对图片就显示得出来。
  文件夹是只读输入，保存 = 下载。

> 对「导出文档 = 一个 md + images 文件夹」的场景，「打开文件夹」是推荐入口。
> 单开一个 md 文件天然够不到 sibling 图片，这是浏览器沙箱限制，不是 bug。
