# 长文档块检索（find-blocks）

日期：2026-07-08
状态：已完成并验收通过（2026-07-08，验收记录见文末）

## 背景与问题

编辑飞书文档时，定位目标块目前只有一条路：`list-blocks` 把整篇文档的块全量列出来，
由调用方（CC）自己在 JSON 里找。这在短文档没问题，长文档下有两个实际痛点：

1. **全量输出代价大**。已验收的 424 块测试文档，`list-blocks` 输出是几百条
   「id + 80 字摘要」，AI 每定位一次都要吞下整篇结构，token 花销和看错行的概率都随
   文档长度上涨。
2. **80 字摘要是盲区**。`summaryOf`（editor.ts）只取块纯文本的前 80 字，要找的内容
   若在长段落的中后部，`list-blocks` 输出里根本看不见，凭内容定位就断了路。

而底层数据其实是现成的：`readTree` 走 client_vars 读管线（2026-07-08 已修好翻页，
拿到的是服务端完整块树，不受 DOM 虚拟化影响），块的完整纯文本 `plainText(b.text)`
就在手边，缺的只是一个「按内容过滤」的出口。

## 目标

新增一个只读命令 `find-blocks`：按关键词（或正则）在**块的完整纯文本**上匹配，
只返回命中的块，输出可直接接续现有块级操作（`--expect` 摘要、块 ID 均可直接用）。

## 非目标（YAGNI）

- 不做模糊匹配 / 相似度排序，子串 + 可选正则够用。
- 不改 `summaryOf` 的 80 字截断——它是 `--expect` 防呆的比对基准，动了会破坏兼容。
- 不返回命中块的邻居块。`insert-after-block` / `replace-block` / `delete-block`
  用命中块自己的 ID 就够；真需要看上下文，再对着 parentId 跑一次 `list-blocks`。
- 不做跨文档搜索。

## 命令接口

```bash
node edit.mjs <链接> find-blocks "<关键词>" [--regex] [--type <类型前缀>] [--limit N]
```

- `<关键词>`：必填，且**归一化去空白后必须非空**（纯空白关键词会让空串子串匹配
  命中所有块，CLI 端就地拦下报 usage 错）。默认按「空白归一 + 忽略大小写」的子串
  匹配（与现有 `normalize()` 同一套空白归一，避免编辑器重排空白导致搜不到）。
  要搜以 `--` 开头的内容，用 `--regex` 加转义绕过参数解析（既有解析器会把 `--`
  开头的位置参数误吞成旗标，不值得为此改解析器）。
- `--regex`：把关键词按 JavaScript 正则解释（正则不做空白归一，原文匹配）。
  CLI 端预检和扩展端执行必须用**完全相同的旗标 `iu`**——`u` 旗标会让一批无旗标下
  合法的写法变非法（未转义的 `{`、孤立的 `\p` 等），不带旗标预检会放过这类正则，
  到扩展端才炸。语法错在 CLI 端报 usage 错，不发请求。
- `--type <前缀>`：按块类型前缀过滤，如 `--type heading` 只搜标题（前缀匹配，
  与 `insert-after` 里 `type.startsWith('heading')` 的判法一致）。
- `--limit N`：最多返回 N 条命中（默认 20）。超出时输出里带 `total` 报告真实
  命中数——宁可明说截断，不可让全量 dump 换个马甲回来。

## 输出格式（stdout JSON）

```json
{
  "ok": true,
  "total": 37,
  "shown": 20,
  "blocks": [
    {
      "id": "doxcn...",
      "parentId": "doxcn...",
      "type": "text",
      "depth": 1,
      "childCount": 0,
      "summary": "该块纯文本前 80 字（与 list-blocks 完全一致，可直接当 --expect 用）",
      "snippet": "…命中处前后各约 40 字的上下文，命中词用【】标出…"
    }
  ]
}
```

两个字段各司其职，缺一不可：

- `summary`：必须与 `list-blocks` 的 `summaryOf` 逐字一致，这样 CC 拿到就能直接
  填给 `replace-block --expect` / `delete-block --expect`，不用再跑一遍 `list-blocks`。
- `snippet`：解决 80 字盲区——命中发生在段落深处时，`summary` 里看不到命中词，
  `snippet` 给出命中位置的局部上下文，供 AI 确认「就是这一块」。取第一处命中，
  前后各约 40 字，超出处用 `…` 标省略。

## 数据流与改动点

数据流与 `list-blocks` 完全同构：CLI → daemon（原样透传）→ 扩展后台 `runEditJob`
→ `readTree` 读服务端块树 → 过滤 → 结果原路返回。**不开编辑流程、不动粘贴链路、
不看 DOM**，虚拟化渲染骗不到它。

### 1. `src/background/editor.ts`（主要改动，约 40 行）

- `EditJob.anchor` 类型扩一档：`{ query?: string; regex?: boolean; typeFilter?: string; limit?: number }`。
  查询参数塞进 `anchor` 而不是新增顶层字段，是刻意的：daemon 对编辑任务只透传
  `kind/op/contentMd/anchor` 四个字段（daemon.mjs 明确写了"不理解语义"），复用
  `anchor` 则 **daemon 零改动**，也不存在"用户机器上还跑着旧 daemon 不透传新字段"
  的版本错位问题。
- `runEditJob` 里紧挨着 `list-blocks` 分支加 `find-blocks` 分支（同属只读，走
  `readTree` 之后、`resolveTarget` 之前）。
- 新函数 `findBlocks(tree, anchor)`：
  1. 复用 `flatten(tree)` 的深度优先顺序遍历（保证命中块按文档顺序输出）；
  2. 对每块取 `plainText(b.text)` 全文匹配——默认路径关键词和全文都过
     `normalize()` 去空白再 `toLowerCase()` 比子串；`regex` 路径直接
     `new RegExp(query, 'iu')` 试原文；
  3. `typeFilter` 用 `b.type.startsWith(前缀)` 过滤；
  4. 命中块产出 `summary`（直接调既有 `summaryOf`）和 `snippet`（在原始纯文本上
     定位首个命中的位置，前后各截 40 字；默认路径因为比对时去了空白，需要一个
     「归一化偏移 → 原文偏移」的小映射，且命中的**起点和终点都要映射**——去空白
     匹配下命中范围在原文里可能夹着空白，只映射起点会把【】标错半截。实现时注意
     别在这里省事直接展示归一化文本）；
  5. 数够 `limit` 条后继续只计数不收集，最终返回 `{ total, blocks }`。
- 埋点照抄 `list-blocks`：`track({ name: 'edit', url: '/edit/find-blocks', ... })`。

### 2. `skills/larksnap-fetch/scripts/edit.mjs`（约 30 行）

- `switch (op)` 加 `find-blocks` 分支：校验关键词必填且归一化后非空；`--regex`
  时先 `new RegExp(query, 'iu')` 预检（旗标与扩展端执行完全一致），语法错就地报
  usage 错；组装 `anchor = { query, regex, typeFilter, limit }`。
- 旗标注册：`--type`、`--limit` 进 `FLAGS_WITH_VALUE`，`--regex` 进 `FLAGS_BOOL`；
  `--limit` 要验证是正整数。
- `handleLine` 的 `result` 分支：`op === 'find-blocks'` 时把
  `{ ok, total, shown, blocks }` JSON 打到 stdout（与 `list-blocks` 同款处理）。
- USAGE_HINT 加一行用法。

### 3. `skills/larksnap-fetch/SKILL.md`（文档）

- 命令清单加 `find-blocks`。
- 把「块级操作先 list-blocks 再动手」的指引改为：**长文档（或已知要找什么内容时）
  优先 `find-blocks` 定位，`list-blocks` 留给需要看整体结构的场景**。
- 说明输出里 `summary` 可直接当 `--expect` 用、`snippet` 只用于人/AI 确认，
  不要拿 `snippet` 去填 `--expect`。

### 4. 不改的东西

- `bridge/daemon.mjs`：零改动（查询参数搭 `anchor` 的便车）。
- `editor-inject.ts`：只读操作不进页面，不涉及。
- `summaryOf` / `normalize`：保持原样，兼容 `--expect` 链路。

## 边界与兼容

- **新 CLI + 旧扩展**：仓库里 CLI 与扩展一起发版，但用户浏览器里可能挂着旧扩展。
  旧版 `runEditJob` 不认识 `find-blocks`，会掉进 `resolveTarget` 的 default 分支
  （editor.ts:396），返回无 subtype 的 `未知编辑操作「find-blocks」`，CLI 端落到
  兜底分类报 `edit_failed`。可接受（只读操作无副作用），CLI 端在 EDIT_ERROR_KINDS
  不用专门建档，但 SKILL.md 里提一句「报『未知编辑操作』先重载扩展」——这个
  message 本身很好认。
- **零命中**：不算错误，正常返回 `{ ok: true, total: 0, blocks: [] }`，让 AI 自己
  决定下一步（换关键词或退回 list-blocks），不要报 `anchor_not_found`——那是写操作
  的语义。
- **非文本块**：`plainText` 为空的块（图片、分割线等）永远不命中。`--type` 只是
  过滤条件，不能单独当「按类型列块」用——那是 `list-blocks` 的活，find-blocks
  不长第二形态（YAGNI）。
- **正则灾难**：用户给了会回溯爆炸的正则，扩展侧跑在 Service Worker 里可能卡死。
  一期不做超时防护（YAGNI，自用工具），但正则统一 `iu` 不加 `g`，单块文本量有限，
  实际风险很小。

## 验收

测试文档沿用长文档修复验收的两个环境（iflytek 私有化 424 块文档 + 公有云文档），
改扩展代码后提醒用户在 chrome://extensions 手动重载扩展再测：

1. **盲区用例（核心）**：在 424 块文档里挑一个长段落，用只出现在第 80 字之后的
   词搜索——必须命中，且 snippet 里能看到命中词、summary 与 list-blocks 一致。
2. **截断报告**：搜一个高频词使命中数 > 20，验证 `total > shown`、blocks 恰好
   20 条且按文档顺序。
3. **类型过滤**：`--type heading` 搜标题词，只回标题块。
4. **正则**：`--regex` 搜一个带量词的模式；再给一个语法错的正则，验证 CLI 端
   直接报 usage 错、不发请求。
5. **接续写操作（端到端）**：`find-blocks` 定位 → 拿输出的 id 和 summary 直接跑
   `replace-block --expect`，一次通过，不再跑 list-blocks。
6. **零命中**：搜不存在的词，返回 `total: 0`，退出码 0。
7. **回归**：短文档六命令（append / insert-after / list-blocks / replace-block /
   delete-block / insert-after-block）+ replace-all 跑一轮确认没碰坏。
